# Code review

How changes are reviewed before landing on `main`. This standard defines what a reviewer looks at, what authors prepare, what constitutes "approved", and when a second reviewer is mandatory.

## Scope

Every change to `main`. During the current solo phase, the maintainer reviews their own work through a rigorous self-review pass; once external contributions begin, this document becomes the contract between author and reviewer.

## Principles

- **Review is a conversation, not a gate.** The goal is to land the best version of a change, not to slow it down.
- **Small changes review better than big changes.** A 200-line PR gets a careful read; a 2000-line PR gets a skim and rubber-stamp.
- **Approving a change is a statement of responsibility.** The reviewer shares ownership of the merged result.

## Author's responsibilities before requesting review

1. **Self-review first.** Read the whole diff as if someone else wrote it. You will find at least one thing worth changing.
2. **Run the local checks.** `cargo fmt --all`, `cargo clippy --workspace -- -D warnings`, `cargo test --workspace`, QEMU smoke if applicable.
3. **Write a good PR description.** Explain the *why*, the approach, the alternatives considered, and anything unusual.
4. **Reference affected ADRs.** If the change implements or affects an ADR, link it. If it would require an ADR but does not have one, write the ADR first.
5. **Flag risk.** If the change touches security-sensitive subsystems, an `unsafe` region, a capability contract, or the boot path, say so in the description and request security review.
6. **Stay small.** If the PR exceeds ~400 changed lines (not counting autogenerated files), consider whether it can be split. A PR that splits cleanly into "plumbing", "logic", and "tests" usually should.
7. **Post-fix sweep.** When a commit corrects a doc typo, renames a symbol, or changes a literal string that may appear in multiple places, the author runs `git grep -F '<old-token>'` (or `rg -F`) before staging and either fixes every match or explicitly justifies leaving the rest. Skipping this step is the failure mode the project rediscovered three times — once during T-009's review-fix cascade, once during the cool-down withdrawal ripple in PR #9, and once during the `ipc.md` broken-link sweep that the second review-pass caught. The pattern is cheap to apply (one grep per fix commit) and it closes the "fix produces stale documentation" anti-pattern that fix-round commits otherwise generate. Especially relevant when the fix touches: filenames cited from docs (e.g. a task-file rename), audit-log prose that may also appear in source comments, ADR-section references (e.g. `§Rule 1` numbering changes), or any literal asm / instruction-encoding text that may appear in both source and `docs/architecture/*.md`. Reviewers verify item 7 by running the same grep themselves on the change's tip.

## Reviewer's responsibilities

### Core reading pass

1. **Read the PR description first.** Understand the intent before looking at the diff.
2. **Look at the tests.** Tests tell you what the author thinks the behavior is. Read them before the implementation; you will often spot design issues earlier.
3. **Read the diff in topological order.** New types → their impls → the callers. Do not read linearly through files; read by dependency.
4. **Check the checklist** (below).
5. **Run the change.** For non-trivial PRs, build locally and, if the change has a behavioral effect, run the QEMU smoke suite. CI is necessary but not always sufficient.

### Review checklist

The reviewer explicitly considers:

- **Correctness.** Does the code do what the description says? Are there off-by-one errors, uninitialized fields, missing error handling?
- **Capability semantics.** If the change touches a trust boundary, is the capability check in the right place? Can an unauthorized caller observe or affect state they should not?
- **Fault containment.** Can a userspace failure reach the kernel through this path? If so, is that intended?
- **`unsafe` discipline.** Any new `unsafe` must satisfy [unsafe-policy.md](unsafe-policy.md): `SAFETY:` comment, audit log entry, justification, reviewer signed off specifically on the `unsafe`.
- **Error handling.** Every `Result` propagated correctly? No new `unwrap`/`expect` on hot paths? Error types converted at boundaries (see [error-handling.md](error-handling.md))?
- **Tests.** Do tests actually test the change? See [testing.md](testing.md) for what counts.
- **Documentation.** Every new public item documented? Every `unsafe fn` has a `# Safety` section? ADR references in commit messages?
- **Architectural principles.** Does the change respect [architectural-principles.md](architectural-principles.md), especially P1 (no ambient authority), P2 (small TCB), P3 (drivers in userspace), P7 (no proprietary blobs)?
- **Dependencies.** Any new crate added? Has it been through the dependency-addition process in [infrastructure.md](infrastructure.md)?
- **Naming and style.** Conforms to [code-style.md](code-style.md)? Clippy would already have caught most of this; reviewer does not re-litigate.
- **Commit message.** Conforms to [commit-style.md](commit-style.md)? Trailers correct?

### Security review gate

A change that touches any of the following requires a **second reviewer** with security context (see [security-review.md](security-review.md) for the full criteria):

- Capability types, capability tables, capability transfer.
- IPC message format, IPC entry points.
- Syscall entries or the syscall ABI.
- Memory management (page tables, TLB, allocators).
- The boot path.
- Anything cryptographic.
- Any new `unsafe` region.

During the solo phase, the maintainer fills both reviewer roles with an explicit second pass — reading the change cold, at least several hours after writing it, with the security checklist in hand.

### What a reviewer does *not* spend energy on

- **Rustfmt output** — CI handles it.
- **Clippy-catchable issues** — CI handles it.
- **Bikeshedding** — "I would have written it this other way but yours is fine" is not a review comment; it is a diary entry.
- **Scope expansion** — the reviewer does not ask the author to "also refactor X while you're in there." Open a separate issue.

## PR size

- **Small:** < 100 changed lines. Reviewed quickly.
- **Medium:** 100 – 400 changed lines. Target size for most PRs.
- **Large:** 400 – 1000 lines. Acceptable with justification.
- **Very large:** > 1000 lines. Strongly discouraged except for:
  - Initial bootstrap commits (like the one establishing docs and ADRs).
  - Autogenerated diffs (lockfile updates, formatter sweeps).
  - Cohesive refactors that would be misleading to split.

The author explains any PR over "Medium" in the description, naming the reason it cannot be smaller.

## Turnaround

- First response (approval, comments, or "reviewing"): 48 hours.
- The author does not block on review for trivial changes to their own branch; they can continue building on it. But merging waits.
- Stale PRs (> 7 days without author activity after review) can be closed pending a refresh.

## Approval semantics

- **Approve:** the reviewer agrees the change is ready to merge. Minor suggestions may remain; the author decides whether to address them before merging.
- **Comment / request changes:** the reviewer identifies issues that must be addressed before merging.
- **Request changes is not punitive.** It is the explicit signal that the PR is not yet mergeable.

## Merge

- Merges are performed by the author after receiving approval, unless the author lacks merge permission.
- Squash and merge by default (see [commit-style.md](commit-style.md)).
- Rebase merge allowed when the PR history is already clean.
- Never merge with CI red. Never merge with unresolved "request changes".

## Self-review (solo phase)

During the current phase, the maintainer practices:

1. Write the change.
2. Open a PR against `main`.
3. Wait. Ideally at least a few hours; ideally overnight for anything substantial.
4. Review the PR from the top — read the description, read the tests, read the diff. Pretend you have never seen it.
5. Merge only after that cold pass.

This is not a substitute for a second set of eyes, but it is a substantial improvement over committing straight to `main`.

## Anti-patterns to reject

- "LGTM" without reading the code.
- Scope expansion comments ("while you're here, also …").
- Re-litigating style the tools already caught.
- Approving to unblock the author without actually approving.
- Authors merging their own PR without review just because review is slow. Slow review is a process problem; fix the process.
- Reviewers holding a PR hostage for preference disagreements.

## Tooling

- GitHub PR reviews.
- `gh pr view`, `gh pr review` — CLI for workflow.
- CI (planned) enforces format, clippy, tests, coverage delta.
- Branch protection (planned) blocks merges without review on `main`.
