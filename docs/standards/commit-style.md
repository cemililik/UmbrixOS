# Commit style

How commits are structured in the Tyrne repository. The goal is a git history that is useful years later — searchable, bisectable, and self-explanatory — without imposing ceremony that obstructs small changes.

## Scope

Every commit to every branch. Local work-in-progress commits are the author's business, but commits that reach `main` (via merge, squash, or direct push during the solo phase) must conform.

## Format

Each commit follows the Conventional Commits pattern with Tyrne-specific trailers:

```
<type>(<scope>): <subject>

<body>

<trailers>
```

Example:

```
feat(ipc): introduce synchronous rendezvous endpoint

Adds a kernel-level endpoint object that rendezvouses one sender with
one receiver. The sender blocks until the receiver issues a matching
receive call. Capabilities to the endpoint are required on both sides.

The non-blocking variant (async notifications) is a follow-up and will
be introduced once the blocking path has been exercised under QEMU.

Refs: ADR-0001
```

### Subject line

- **Type** — one of:
  - `feat` — a user-visible feature addition.
  - `fix` — a bug fix.
  - `docs` — documentation changes only.
  - `refactor` — code change that neither fixes a bug nor adds a feature.
  - `perf` — a performance improvement with a measurement.
  - `test` — adding or correcting tests.
  - `build` — build system, dependency updates.
  - `ci` — continuous integration configuration.
  - `chore` — tooling, reformatting, licensing housekeeping.
  - `revert` — reverts a previous commit (subject: `revert: <original subject>`).
- **Scope** — the subsystem most affected. Choose from the short list:
  `kernel`, `hal`, `ipc`, `mm`, `sched`, `syscalls`, `drivers`, `userspace`,
  `bsp-<board>`, `docs`, `standards`, `adr`, `ci`, `tooling`, `deps`. New scopes
  must appear in at least one other prior commit or be agreed in review.
- **Subject** —
  - Imperative mood ("add", not "adds" or "added").
  - Lowercase first word (type takes care of emphasis).
  - No trailing period.
  - ≤ 72 characters. Strongly prefer ≤ 60 for scannability.

### Body

- Blank line after the subject.
- Explains **why**. The diff shows the *what*.
- Wrap at ~100 characters for terminal readability.
- Use bullets where they help; full sentences otherwise.
- Do not paste the diff.

When a commit is trivially self-explanatory (`docs(glossary): fix typo`), the body may be omitted.

### Trailers

Use `key: value` trailers at the bottom of the body. Each trailer is on its own line; a blank line separates the body from trailers.

- `Refs: ADR-NNNN` — references the ADRs a change implements or relates to. Required for any commit that implements, extends, or affects an ADR-governed subsystem.
- `Fixes: #<issue>` — the commit closes a GitHub issue.
- `Closes: #<issue>` — alternative to `Fixes` for non-bug issues.
- `Audit: UNSAFE-YYYY-NNNN` — references the audit entry for any commit that introduces or modifies an `unsafe` block. See [unsafe-policy.md](unsafe-policy.md).
- `Security-Review: <reviewer>` — for commits touching security-sensitive subsystems, records the second reviewer. See [security-review.md](security-review.md).
- `Co-authored-by: Name <email>` — optional; used when a commit has multiple authors (including AI agents, if the author chooses to credit).
- `Signed-off-by: Name <email>` — Developer Certificate of Origin sign-off if the maintainer enables that workflow (not required in the current phase; to be revisited).

Trailers that reference missing objects (a nonexistent ADR, an issue with the wrong number) are a review-rejection reason.

## Granularity

- **One logical change per commit.** A commit that touches the scheduler, the IPC path, and the docs is three commits.
- **Small commits beat big commits.** A reviewable commit is typically < 400 changed lines, not counting autogenerated files.
- **Bisectable.** Every commit on `main` should build, pass the fast test suite, and be a meaningful checkpoint. Squashing at merge time is used to enforce this.

Exceptions:

- The **initial bootstrap** (this phase) may include many files in one commit because the repo was empty and the files are mutually referential.
- **Autogenerated changes** (e.g. `cargo update` that changes many `Cargo.lock` lines) may stand alone even when large.

## Merging strategy

- **Squash + linear history** is the default for PRs when the PR history is messy (multiple WIP commits).
- **Rebase merge** is acceptable when the PR's commit history is already clean and each commit is individually valuable.
- **Merge commits** are not used on `main`. The history is linear.

## AI-authored commits

When an AI agent produces a commit on the maintainer's behalf:

- The commit author is the maintainer (the agent is a tool).
- A `Co-authored-by:` trailer may credit the agent at the maintainer's discretion.
- The commit message follows this standard exactly — no agent-specific suffix, no auto-generated "Generated with …" line.
- Any commit that introduces `unsafe`, crosses a trust boundary, or adds a dependency requires explicit maintainer review before being committed by the agent.

## Anti-patterns to reject

- `fix: stuff`, `wip`, `changes`, `merge`, `update` — uninformative subjects.
- Subjects that describe the *how* ("change `x` to `y`") rather than the *why* or *effect*.
- Bodies that restate the diff in prose.
- Commits that bundle feature + unrelated refactor + typo fix.
- Trailers that lie: `Refs: ADR-0099` when ADR-0099 does not exist.

## Tooling

- `.gitmessage` template (to be added) — pre-populates the subject/body structure on `git commit` when configured via `git config commit.template`.
- Commitlint-style CI validation — planned, once PRs start flowing.
- `git log --oneline --decorate --graph` should be readable at a glance; if it is not, history is drifting.
