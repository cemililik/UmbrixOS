# Performance baseline 2026-04-28 — B1 closure (post-T-013 + T-012)

- **Concern:** Did T-013's EL drop + T-012's GIC v2 driver, EL1 vector table, and generic-timer IRQ wiring produce measurable overhead in image size, kernel RAM footprint, or hot-path instruction counts versus the [A6 baseline](2026-04-21-A6-baseline.md)?
- **Scope:** The full B0+B1 arc — every code change committed between the A6 release tag (`52cd72f` per the A6 baseline) and the PR #10 merge to `main`. This includes T-006 / T-007 / T-008 / T-009 / T-011 (B0) plus T-013 / T-012 (B1). Per-milestone decomposition is performed where the section symbol or function boundary makes it possible; bundle attribution is given where it is not.
- **Hypothesis:** This is a re-baseline artifact, not a hypothesis-driven optimization cycle. Per the [master plan's pre-flight](master-plan.md#pre-flight-hypothesis), no concrete improvement target is set — the goal is to record the v0.0.2-track baseline so future hypothesis-driven cycles (preemption, `time_sleep_until` latency, IPC round-trip with the now-available `now_ns()`) have a reference point. The implicit non-hypothesis: structural overhead from B0+B1 should not exceed ~5–10 KiB in the kernel image and ~5 KiB in `.bss`, both upper bounds derivable from the new code surface (vector table 2 KiB, GIC driver ~1.5 KiB, idle stack 4 KiB).
- **Reviewer:** @cemililik (+ Claude Opus 4.7 agent acting in the Baseline + Hotspot + Reporter roles below; Proposal / Measurement / Regression-check are minimal because no proposal is being measured this cycle).
- **Target:** QEMU `virt`, aarch64, Cortex-A72 model, single core, 128 MiB RAM (unchanged from A6).
- **Build:** release profile (`cargo build --release --target aarch64-unknown-none -p tyrne-bsp-qemu-virt`).

## Baseline

### Methodology

- ELF section sizes via `rust-readobj --elf-output-style=GNU -S` (workspace `cargo` toolchain pin).
- Stripped flat binary via `rust-objcopy --strip-all -O binary`.
- Kernel image builds at the workspace's `release` profile (the same profile the A6 baseline used).
- Static instruction counts derived by hand from the source — the QEMU emulation does not give meaningful wall-clock numbers, and no real-hardware target is plumbed yet (Pi 4 is Phase D; per A6 baseline §Metric 4).
- Single-run measurements; no iteration / variance recorded because the build is deterministic and the section sizes are byte-stable across rebuilds.

### Metric 1 — Kernel image size

| Section | A6 (2026-04-21) | B1 (2026-04-28) | Δ bytes | Δ % |
|---------|----------------|-----------------|---------|-----|
| `.text` | 13,940 (13.6 KiB) | 21,908 (21.4 KiB) | +7,968 | +57.2 % |
| `.rodata` | 1,960 (1.9 KiB) | 2,784 (2.7 KiB) | +824 | +42.0 % |
| `.bss` | 17,872 (17.5 KiB) | 22,248 (21.7 KiB) | +4,376 | +24.5 % |
| **Stripped flat binary (`-O binary`)** | 80 KiB (per A6 report) | **24,704 (24.1 KiB)** | (see note) | (see note) |

> **Note on the flat-binary discrepancy.** The A6 report records 80 KiB for the stripped flat binary; the B1 measurement is 24.1 KiB. The two are not directly comparable: the flat-binary size depends on how the linker-defined LOAD segments are laid out, and a `BSS-as-zero-padding` LOAD segment would inflate the A6 number while a `BSS-as-NOBITS` segment (today's case, per the [`linker.ld`](../../../../bsp-qemu-virt/linker.ld) `.bss : ALIGN(8) { ... } > RAM` construction) would not. The section-level breakdown above is the canonical comparison and is byte-accurate for both measurements. **Recommendation:** drop the flat-binary line from future baseline reports unless an audit step recomputes the A6 number under the current linker script. The per-section figures are sufficient and unambiguous.

**Decomposition of `.text` growth (+7,968 bytes).** The new code shapes break down as:

Includes the `PanicClass::from_u64` dispatch + format machinery; `irq_entry`'s straight-line ack-and-ignore body is small. |
| `gic.rs` (driver) | ~1,600 | Constructor, `init`, four trait methods, four MMIO helpers, `Send`/`Sync` markers. The GICD_TYPER-bounded init loops dominate. |

| `main.rs` GIC + VBAR + DAIF setup | ~140 | Three `unsafe` blocks in `kernel_entry` between the timer-banner block and the kernel-object setup block. |
| `boot.s` EL drop (T-013) | ~80 | 20 instructions of EL2→EL1 transition. |
| `hal/src/cpu.rs::current_el` (T-013) | ~50 | Single `MRS CurrentEL` + mask. |

**B1-specific contribution (T-013 + T-012 only): ~4,636 bytes** = 2,048 (vector table) + 168 (trampolines) + 244 (exceptions.rs, objdump-verified) + 1,600 (gic.rs) + 280 (cpu.rs Timer arm/cancel) + 140 (main.rs setup) + 80 (boot.s EL drop) + 50 (current_el) + 26 (hal::timer `ns_to_ticks` half — T-012's new helper, not the T-009 `ticks_to_ns` half). 4,636 / 7,968 ≈ **58 %** of the A6→B1 `.text` delta is T-013 + T-012 specifically; T-013 alone adds ~130 bytes; T-012 alone adds ~4,506 bytes; B0 contributes the remaining ~42 %.

The 4 KiB-per-task stack budget remains the single largest line item. Reducing `TASK_IDLE_STACK` to 1 KiB once the watermark is measured would recover ~3 KiB; deferred to the first hypothesis-driven RAM-reduction cycle.

### Metric 3 — IPC round-trip latency

**Still not measured at wall-clock granularity.** The A6 baseline noted that `now_ns()` was unavailable in Phase A. T-009 lit up `CNTVCT_EL0` reads at EL1, so `Cpu::now_ns()` is now usable from kernel code; the infrastructure to measure IPC round-trip exists, but no benchmark harness has been built. **Static instruction-count estimate (one round trip — task A sends, task B receives, B replies, A receives reply):**

| Operation | A6 estimate | B1 estimate | Δ |
|-----------|-------------|-------------|---|
| `context_switch_asm` (save+restore) | 27 instructions × 2 switches = 54 | unchanged at 27 × 2 = 54 | 0 |
| `ipc_send` state machine | ~30 | ~30–35 | +0–5 (T-006 raw-pointer bridge added a momentary-`&mut` materialisation; release-profile inlining absorbs most of it) |
| `ipc_recv` state machine (×2) | ~60 | ~62–70 | +2–10 |
| `unblock_receiver_on` (O(N), N=2 tasks for the demo) | ~10 | ~10 | 0 |
| `IrqGuard` (DAIF mask + restore) | 8 | 8 | 0 |
| **Approximate total** | **~162** | **~166–177** | **+0–9 %** |

The T-006 raw-pointer bridge introduces a momentary `&mut Scheduler<C>` discipline whose materialisation cost is bounded by one `ldr` (raw-pointer-to-`&mut` conversion) + one drop-of-`&mut` (free); release-profile inlining typically folds both into surrounding scheduler code. The 0–9 % range is the upper-bound cost; an actual measurement would likely show <2 %. Recommendation captured below.

### Metric 5 — Boot time (reset → "all tasks complete")

| Phase | A6 | B1 |
|-------|------|------|
| `_start` (asm) zero BSS | ~2,240 STR XZR iterations | ~2,780 iterations (`.bss` grew 4.2 KiB; +540 iterations) |
| EL drop (T-013) | — | ~10 instructions on the EL2 path (skipped on EL1 entry) |
| Vector table install | — | 4 instructions (adrp + add + msr + isb) |
| GIC init (`gic.init()`) | — | ~1,600 instructions (depends on `GICD_TYPER`-reported line count; for 1,024 lines: ~248 ICENABLER + ~248 IPRIORITYR + ~248 ITARGETSR iterations) |
| DAIF unmask | — | 1 instruction |
| Kernel-object setup | ~few hundred | ~few hundred + ~15 for idle task allocation |
| Timer-banner + boot-ns capture (T-009) | — | ~30 instructions |
| **Estimated reset → kernel_entry first instruction** | < 1 ms | < 1 ms (at QEMU; ~1.7 µs additional on real Cortex-A72 @ 1.8 GHz / 1 IPC) |
| **Estimated reset → "all tasks complete"** | < 1 ms (imperceptible at QEMU wall clock) | < 1 ms (imperceptible) |

The ~1,600 instructions of GIC init are the dominant new cost on the boot path. At a real Cortex-A72 1.8 GHz with L1 cache hits, ~900 ns; on QEMU emulation, indistinguishable from prior boot. The bound is structurally tight to ARM IHI 0048B §4 (architectural init sequence); the only optimization opportunity is `gic.enable(specific_irq)` style on-demand init for individual lines, which v1 doesn't need.

### Metric 6 — IRQ delivery cost (new — T-012 introduction)

This metric did not exist at A6 (no IRQ infrastructure). Static instruction counts for the timer-IRQ path:

| Phase | Instructions | Notes |
|-------|--------------|-------|
| Hardware: take exception, save ELR_EL1 + SPSR_EL1 to banked regs, branch to VBAR_EL1+0x280 | (microcoded) | Cortex-A72 architectural cost ~10–20 cycles. |
| Trampoline save (`sub sp` + `stp` × 10 [GP pairs] + `mrs elr` + `mrs spsr` + `stp` × 1 [elr/spsr pair] + `mov x0, sp`) | 15 | Verified line-by-line against [`vectors.s:115-130`](../../../../bsp-qemu-virt/src/vectors.s); the `mov x0, sp` writes the frame-pointer parameter for `bl irq_entry` (not `mov sp, x0` — that would corrupt SP). |
| `bl irq_entry` | 1 | Standard branch-with-link. |
| `irq_entry` body (timer-IRQ path) | ~20 | Read GIC_IAR + spurious check + IRQ ID dispatch + MSR CNTV_CTL_EL0 + write GICC_EOIR + return. |
| Trampoline restore (`ldp` × 1 [elr/spsr pair] + `msr elr` + `msr spsr` + `ldp` × 10 [GP pairs] + `add sp` + `eret`) | 15 | Verified line-by-line against [`vectors.s:133-147`](../../../../bsp-qemu-virt/src/vectors.s); mirror of save with the SP restore folded into the literal `add sp, sp, #192`. |
| Hardware: consume ELR/SPSR, return to interrupted PC | (microcoded) | Cortex-A72 architectural cost ~5–10 cycles. |
| **Approximate dispatch cost (one timer IRQ, ack-and-ignore)** | **~51 instructions + ~25 cycles of microcode** | Estimated 35–50 ns on Cortex-A72 @ 1.8 GHz. |

For comparison, the [A6 baseline §Metric 3](2026-04-21-A6-baseline.md) estimated IPC round-trip at ~162 instructions; one IRQ dispatch is ~⅓ of an IPC round trip in static count. Whether that maps to wall-clock proportionally depends on cache locality, which only a real measurement can determine.

**v1 does not arm a deadline** — the IPC demo's `task_a` / `task_b` make no Timer-trait call; idle's body is structurally unreachable in the demo path. So the IRQ delivery cost is *available* but *not exercised* in v1. The measurement is for future preemption / `time_sleep_until` arcs.

## Hotspot

For an ack-and-ignore IRQ path, the trampoline save+restore is the dominant cost (30 of 51 instructions = ~59 %). Two structural observations:

1. **Trampoline save+restore is bounded by AAPCS64 caller-saved set.** The trampoline saves x0..x18 + x30 (the AAPCS64 caller-saved general-purpose registers) plus ELR/SPSR. Reducing this to "only the registers `irq_entry` actually clobbers" is unsafe in v1 because `irq_entry` is `extern "C"` and the compiler is entitled to clobber any caller-saved register. The full save is correct.
2. **`TrapFrame` allocation (192 bytes on the stack) is unconditional.** v1's `irq_entry` body never reads the frame (the `_frame: *mut TrapFrame` parameter is unused). A future preemption arc that wants to redirect `ELR_EL1` on resume needs the frame; a v1 ack-and-ignore handler does not. **Optional optimization (deferred):** a slimmer trampoline that saves only ELR + SPSR + a couple of scratch registers would shave ~22 of 30 trampoline instructions for the ack-and-ignore path. **Rejected for v1**: would require a second trampoline shape that the future preemption arc would replace, doubling churn. Re-evaluate when preemption lands and the actual IRQ rate is measurable.

For the boot-path hotspot, `gic.init()` is the dominant new cost (~1,600 of ~2,000 new boot instructions). This is bounded by the ARM IHI 0048B init sequence and not compressible without skipping defensive programming (e.g. trusting firmware to leave SPIs disabled). The current shape is correct.

For the IPC round-trip hotspot, T-006's raw-pointer bridge added a momentary-`&mut` materialisation cost in the 0–9 % range. Without an actual measurement, no specific optimization is justified.

## Proposal

**No proposal made this cycle.** This is a baseline artifact, not a hypothesis-driven cycle. Three specific concerns are recorded for future hypothesis-driven cycles:

1. **IPC round-trip wall-clock measurement.** The infrastructure to measure (T-009's `now_ns()`) exists; the harness does not. Build a kernel-side benchmark that loops `ipc_send_and_yield` ↔ `ipc_recv_and_yield` N times, prints `(now_ns - start_ns) / N`. Compare against the ~166–177-instruction estimate; reconcile. Trigger: maintainer interest, or first preemption-related arc that needs the number.
2. **Stack high-water-mark probes.** Each `TaskStack` is 4 KiB; idle's body is "WFI + yield_now", which uses far less. Add a debug-mode probe that fills the stack with a sentinel byte at task creation, then reads back to measure peak depth. Recover unused KiB by reducing per-task stacks once the empirical figure is known. Trigger: B-late RAM-reduction cycle, or first non-QEMU BSP where RAM is constrained.
3. **`TrapFrame` slimming for ack-and-ignore IRQ handlers.** Save only ELR + SPSR + 2 scratch registers (~10 instructions instead of ~31) when the handler is known to clobber a small set. Rejected for v1 (deferred to preemption introduction); flagged here so the cost is on record.

## Verdict

**Baseline recorded — not a Merge / Reject / Iterate cycle.** Per the [master plan's pre-flight](master-plan.md#pre-flight-hypothesis): a review without a hypothesis is exploration, logged as a baseline artifact. This artifact is the v0.0.2-track baseline — kernel image: 21.4 KiB `.text` + 2.7 KiB `.rodata` + 21.7 KiB `.bss`; 24.1 KiB stripped flat binary; 27-instruction context switch; ~51-instruction timer-IRQ dispatch (new — 30 trampoline + 1 bl + 20 handler body); ~166–177-instruction IPC round-trip estimate (mostly unchanged from A6); idle WFI activated (structurally unreachable in the v1 cooperative IPC demo).

**Three concerns flagged for future hypothesis-driven cycles** (not blocking B1 closure):

1. **IPC round-trip wall-clock measurement.** Build a kernel-side benchmark using T-009's `now_ns()`. Compare against the static-count estimate.
2. **Stack high-water-mark probes.** Empirical per-task stack usage; recover unused per-task RAM if the figures justify it.
3. **`TrapFrame` slimming for ack-and-ignore handlers.** Defer until preemption lands and the IRQ rate is measurable; flagged so the cost is on record.

**Pre-closure work-items (gating B1 → Done at the milestone level rather than at the task level):**

- Maintainer-side QEMU smoke verification of the deliberate-deadline path. Lifts the `Pending QEMU smoke verification` status notes on UNSAFE-2026-0019 / 0020 / 0021 via append-only Amendment.
- `cargo +nightly miri test --workspace --exclude tyrne-bsp-qemu-virt` on R6 CI skeleton.

Surface ready for B2 (MMU activation, kernel-half mapping) work to land on top of the now-IRQ-aware kernel.
