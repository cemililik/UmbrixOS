- **Date:** 2026-04-21

- **Purpose:** Record the v0.0.1 baseline so that future reviews have a reference point. No optimization targets are set here тАФ this is measurement, not optimization.

## Metric 1 тАФ Kernel image size (stripped release ELF)

| Item | Value |
|------|-------|
| Stripped binary (flat image, `cargo objcopy --strip-all`) | **80 KiB** |
| `.text` section | 13,940 bytes (13.6 KiB) |
| `.rodata` section | 1,960 bytes (1.9 KiB) |
| `.bss` section (zero-init static storage) | 17,872 bytes (17.5 KiB) |

**Methodology:** `rust-readobj --elf-output-style=GNU -S` on the release ELF; `cargo objcopy --strip-all` for the flat binary size.

**Commentary.** The `.bss` budget (17.5 KiB) is dominated by two 4 KiB task stacks (8 KiB total), one `Scheduler<QemuVirtCpu>` (тЙИ3.3 KiB at 168-byte `Aarch64TaskContext` ├Ч 16 slots + overhead), two `CapabilityTable`s (тЙИ2.6 KiB each at 64 slots ├Ч entry size), and one `IpcQueues` (тЙИ1.0 KiB). This is structurally bounded by the compile-time arena and table capacities; the kernel cannot grow `.bss` at runtime.

## Metric 3 тАФ IPC round-trip latency

**Not measured in A6.** Measuring IPC latency requires a free-running timer readable at EL1. The system counter (`CNTPCT_EL0`) needs the `CNTKCTL_EL1.EL0PCTEN` bit set, and a counter frequency (`CNTFRQ_EL0`) must be initialised тАФ neither is done in Phase A. Phase B will add timer infrastructure.

**Instruction-count estimate (send/reply round trip):**

| Operation | Instruction count (approx.) |
|-----------|---------------------------|
| `context_switch_asm` (save + restore) | 26 STP/LDP/STR/LDR + RET = 27 instructions ├Ч 2 switches = 54 |
| `ipc_send` state machine | тЙИ 30 instructions (validate cap, peek state, replace state) |
| `ipc_recv` state machine (├Ч2, once for each direction) | тЙИ 30 ├Ч 2 = 60 instructions |
| `unblock_receiver_on` (O(N) scan, N=2) | тЙИ 10 instructions |
| `IrqGuard` (2 ├Ч MSR DAIFSET/restore) | 4 instructions ├Ч 2 = 8 |
| **Approximate total** | **~162 instructions** |

At QEMU's emulated Cortex-A72 (not real silicon; throughput is not representative), this is meaningless for wall-clock estimation. The instruction count is provided as a structural baseline for comparing future refactors.

## Metric 4 тАФ Context-switch overhead

**Not measured with a timer** (same reason as metric 3).

**Static instruction count:** 13 `STP`/`STR` instructions (save) + 13 `LDP`/`LDR` instructions (restore) + `RET` = **27 instructions** per switch.

Saved registers: x19тАУx28 (10), x29/fp (1), x30/lr (1), sp (1), d8тАУd15 (8) = 21 registers, 168 bytes per `Aarch64TaskContext`.

On a real Cortex-A72 at 1.8 GHz with L1 cache hits, 27 instructions would complete in roughly 10тАУ20 ns. On QEMU the number is not meaningful. A Phase B review using `CNTPCT_EL0` will produce a real figure.

## Metric 5 тАФ Boot time (reset to `kernel_main`)

**Not measured precisely.** The boot sequence is:

1. `_start` (assembly): set sp, enable FPEN, zero .bss тЖТ тЙИ 10 instructions + .bss zero loop (17.5 KiB / 8 bytes per `STR XZR` = тЙИ2,240 iterations).
2. `kernel_entry` (Rust): construct `Pl011Uart`, `QemuVirtCpu`, publish statics, set up one endpoint, two capability tables, two task objects, scheduler тЖТ тЙИ a few hundred instructions.

Under QEMU the complete sequence from reset to "hello from kernel_main" on the serial port is imperceptible (< 1 ms wall clock). A Phase B review on real hardware (RPi4) should measure this with a UART timestamp.

## Summary

| Metric | Value | Notes |
|--------|-------|-------|
| Stripped image (flat binary) | 80 KiB | Release build |
| `.text` | 13.6 KiB | Code only |
| `.bss` (kernel RAM) | 17.5 KiB | Stacks + kernel state |
| IPC round-trip latency | not measured | Timer not initialised in Phase A |
| Context-switch overhead | 27 instructions | ~10тАУ20 ns on real A72; meaningless on QEMU |
| Boot time (reset тЖТ kernel_main) | < 1 ms on QEMU | Not measured on hardware |

**Verdict.** Numbers are plausible for a v0.0.1 bare-metal kernel with two tasks and no allocator. The `.bss` budget is dominated by statically-bounded arenas, which is expected and correct. No optimization action is triggered by this review; future reviews should add timer-based IPC latency and stack-watermark measurements.
