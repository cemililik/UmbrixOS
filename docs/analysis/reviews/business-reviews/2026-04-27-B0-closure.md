## What we learned

### Cost-of-arc analysis: Phase A vs B0

| Arc | Tasks | Days | Commits | Tests at close | Notes |
|-----|-------|------|---------|---------------|-------|
| Phase A (T-001..T-005) | 5 | 2026-04-20 → 2026-04-21 (~2 days, intense) | ~30 | 109 | Greenfield; scope was the kernel's first end-to-end IPC. Each task closed in 1–2 commits. |
| B0 (T-006..T-011) | 5 | 2026-04-21 → 2026-04-27 (~6 days) | 50+ | 143 | Hygiene + meta-process emergence (ADR-0025). Each task averaged 5–10 commits including review fix-rounds. |

**B0 was 3× the wall-clock of Phase A for the same nominal task count.** The extra cost is concentrated in three places:

1. **ADR-0025's emergence (~1 day)** — a meta-process arc that produced no kernel code but established the rules every later ADR will follow. One-off cost; payoff is forward.
2. **Review fix-rounds (~2 days across PR #9)** — two independent review-pass agents flagged 14 + 5 = 19 findings; two fix-round commits closed them. Plus a third review pass that flagged 5 more. The fix-rounds catch real bugs (the `drop_first_child` test branch fix moved cap/table.rs from 97.46 % to 97.60 % regions because it was actually testing the wrong branch) but each round costs a session.
3. **Audit-log discipline establishment (~0.5 day)** — UNSAFE-2026-0017 and 0018 went through two Amendment rounds to land their final form; this established the introducing-commit boundary as the body-locking event, codified in UNSAFE-2026-0017's "Discipline note for future readers" paragraph. One-time cost; future PRs will have the rule clear.

**Is the 3× ratio sustainable for B1 / B2?** No, and not without intent. B0 was unique in being hygiene-focused — closing review follow-ups, codifying meta-rules, writing architecture docs that summarise prior work. B1 / B2 are forward-feature work; the meta-process cost has been paid (ADR-0025 is in force; audit-log discipline is established; review-fix-round cadence is calibrated). Expect B1 to track closer to Phase A's pace. **The cost-of-arc question for the next retro:** does T-012 land in 1 commit (Phase-A pace), 5 commits (B0 average), or somewhere in between?

### ADR-0025 §Rule 1 (forward-reference contract) — first empirical run

**Outcome: rule fired correctly, zero post-Accept ADR riders during PR #9.**

ADR-0024 was the first ADR to use the *Dependency chain* section in production. Its dependency-chain entry forced T-013 to be opened *as part of the same commit that landed ADR-0024*; a "future, not-yet-opened task X will do Y" handwave was structurally impossible. Result: T-013 had a real T-NNN file before ADR-0024's reasoning relied on it; ADR-0024 went Proposed → Accepted same-day with no rider added before, during, or after PR #9.

This is the rule working exactly as intended. The previous Phase A → B0 arc rediscovered four times that ungrounded forward-references rot — ADR-0021 mid-revision, ADR-0021 post-Accept rider, ADR-0022 first rider, ADR-0022 sub-rider. ADR-0024 is the first ADR not to repeat the pattern.

### ADR-0025 §Rule 2 (rider rate as signal) — calibrating the threshold

**Outcome: still calibrating. ADR-0024 picked up zero riders in its first ~12 hours; we should re-measure at one week.**

The rule frames "3+ riders in a week" as the audit threshold. The B0 arc produced ADR-0021 with 1 rider (post-Accept revision, then sub-rider) and ADR-0022 with 2 riders (first rider + sub-rider). ADR-0024 has 0 so far. The rule's signal value emerges only over a longer window, so this retro flags it as an open empirical question. **Re-measure the rider rate per ADR at the next milestone close.**

### Audit-log Amendment discipline — codified

**Outcome: introducing-commit boundary now explicit; "merge boundary" interpretation explicitly rejected.**

The asymmetry inside PR #9 — V13/V14 went via Amendment but V7 (HCR_EL2 paragraph) went in-place — surfaced a genuine policy gap. The third review-fix round (commit `9680e41`) settled it: §3's "must not be rewritten once committed" means the introducing-commit boundary, not the merge-to-main boundary. This was codified inline in UNSAFE-2026-0017's new "Discipline note for future readers" paragraph. The strict interpretation matches UNSAFE-2026-0006's existing precedent (Amendment for the post-T-009 struct shape). The pragmatic alternative ("same-PR mutable until merge") was deliberately rejected because it would create a second ambiguity (PR merge ≠ commit; not observable from inside the audit log).

**Practical implication for future PRs:** any post-introduction body change to an audit-log entry — typo correction, scope expansion, additional invariant, additional rejected alternative — goes via an Amendment block, regardless of whether the change lands before or after the introducing PR merges to main. Inline body edits to a committed entry's body are forbidden.

### Reviews-as-pattern: now reproducibility-shaped

The independent approval-review agent (introduced 2026-04-27 to verify T-006/T-007/T-009 promotion) ran successfully twice on PR #9 — once for the implementation tip, once for the post-fix tip. The pattern is real: an external reviewer, briefed via a structured prompt that names what's settled vs. what to scrutinise, finds genuine bugs (the `drop_first_child` test wrong-branch case being the most instructive). This has reached sample-size 3+; canonising it as `.claude/skills/conduct-approval-review/SKILL.md` is now warranted as a B1-or-B2 deliverable.

### File renames have a long tail (still)

The T-009 mini-retro flagged this. T-008's review still hit it: four broken cross-references in `ipc.md` (T-003 path × 3, lib.rs vs main.rs) needed the second review-fix round to catch. The post-fix sweep DoD adjustment (still deferred per the mini-retro) is now empirically the third data point — the rule is confirmed and ready to codify.

### Test fixes can be coverage wins

T-011's `drop_first_child` test was passing while testing the wrong branch (mid-list, already covered). Renaming-and-retargeting it to drop the actual list head moved cap/table.rs from 97.46 % to 97.60 % regions. **Lesson:** when a test reaches an unexpected coverage outcome (in either direction), audit the test's actual behaviour against its name and comment. A test that passes for the wrong reason is worse than a test that fails.
