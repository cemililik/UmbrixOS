## What we learned

**Rules work better than re-discovered lessons.** Across A → B0, the same dependency-analysis weakness surfaced four times (ADR-0021 mid-proposal revision, ADR-0021 post-Accept rider, ADR-0022 first rider, ADR-0022 first-rider sub-rider). T-006 mini-retro had named the lesson — "trace the call graph" — but the lesson stayed at the prose level, didn't ship as a rule. T-009's arc reproduced the failure once more (ADR-0022 first rider conflated time-source with IRQ delivery), and only then did we sit down and codify the rule. **The practical implication:** lessons-as-prose tend to be re-discovered; lessons-as-rules in ADR-0013 / template / skills tend to fire on the next attempt. We have not yet validated the codified rules — ADR-0024 is the validation event. Until then, the rules are paper, not progress.

**Reviews left their own stale documentation behind.** The five findings in the third-round review (commit `328cab5`) were almost entirely *generated by* the second-round fix commits (`39fb66c`, `db3a4c7`, `01010f4`). Each fix commit corrected one type of staleness while leaving another type behind: the second-round renamed `cntpct → cntvct` in code but not in task-file title; the runtime-EL-check commit added the assertion but not the doc-comment audit-tag note; the explicit-panic commit added asserts but missed the resolution-floor edge case. The pattern is **mini-rider chains within the rider response itself** — one level of recursion below the ADR/rider chain we already knew about. The third-round was the maintainer's own catch ("are all accepted findings actually fixed?"), not an agent-driven find. **The implication:** even with a strong review process, expect the *fixes* to need a small follow-up sweep one round after they land. Build that into DoD rather than be surprised by it.

**User-driven thoroughness beats agent-driven thoroughness when the agent has been at it for a while.** The maintainer's question — *"have all accepted findings actually been fixed?"* — caught the silently-skipped runtime EL check — a real correctness gap. The agent had self-reported "all accepted findings fixed"; the maintainer's verification surfaced that this was wrong. This is the same lesson as the T-006 retro's "post-In-Review second-read" gate, applied one level deeper: the *fix-up commits* themselves need a verification gate. The user is providing this manually today; whether to automate it (e.g. an "all-findings-actually-applied" agent pass) is open.

**File renames have a long tail.** Renaming `T-009-timer-init-cntpct.md → -cntvct.md` required updates in 4 other files (phase-b/README.md, phase-b.md, current.md, T-012). Without `grep -rln` discipline this is easy to miss. We should normalize: any file rename in `docs/analysis/tasks/` or `docs/decisions/` triggers a `grep -rln <old-slug>` step before commit. Cheap; would have caught two of the third-round review's findings preemptively.
