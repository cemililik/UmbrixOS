## What we learned

### Cost-of-arc analysis: B1 vs B0 vs Phase A

| Arc | Tasks | Days (wall clock) | Commits | Tests at close |
|-----|-------|-------------------|---------|---------------|
| Phase A (T-001..T-005) | 5 | ~2 (2026-04-20 → 2026-04-21) | ~30 | 109 |
| B0 (T-006..T-011) | 5 | ~6 (2026-04-21 → 2026-04-27) | 50+ | 143 |
| **B1 (T-013 + T-012)** | **2** | **~2 (2026-04-27 → 2026-04-28)** | **~14** | **149** |

**B1's pace is ~7 commits/day — between A's 15/day and B0's 8/day.** The B0 retro's hypothesis was "expect B1 to track closer to Phase A's pace; meta-process cost has been paid". Empirical answer: **partially right.** B1 *was* faster than B0 per commit; the meta-process cost (ADR-0025, audit-log discipline) genuinely paid forward. But B1 was *not* as fast as Phase A because:

1. **PR #10 review-fix-rounds cost two commits.** Round-1 (`d820a88`) closed 11/12 findings; round-2 (`5c9cf06`) closed 8 findings + 2 nitpicks. Each round catches a class of bug a casual review misses (round-1 was code: `unsafe extern "C"`, `GIC_MAX_IRQ` range assertion, `ns_to_ticks` floor-vs-ceiling rounding; round-2 was discipline: SAFETY-comment triplets, ADR-amendment dates, prose matching shipped behaviour). Both rounds were necessary; neither was redundant. **Lesson: the review-fix-round cadence is now a load-bearing part of the velocity story; it's not free, and it shouldn't be.**
2. **Design-first arc cost one commit (`c366200`).** The architecture doc landed first, then implementation. This *saved* commits later (no "doc follow-up" commit was needed; the documentation-sweep commit `28c5ce9` was about ADR Revision-notes + cross-references, not the design doc). **Net effect: probably break-even on commit count, but distributed differently — front-loaded rather than back-loaded.**
3. **Maintainer-side gates don't appear in commit count.** The `Pending QEMU smoke verification` status notes mean the QEMU smoke + Miri runs happen *after* the merge, not as a pre-merge gate. This shifts work outside the development-environment commit loop. **Lesson: the development-environment velocity number undercounts total milestone work — the smoke + Miri runs are real cost, just not visible in `git log`.**

**Is the ~7 commits/day pace sustainable for B2?** Probably yes. B2 is a forward-feature arc (MMU activation) with a similar shape to B1 (one substantial implementation task, possibly preceded by an ADR + design doc). The review-fix-round cost will recur but is bounded. **Anticipation for B2: ~10–15 commits across the MMU-activation arc, possibly split into ADR-first → page-tables → activation sequence → tests.**

### Audit-log "Pending QEMU smoke verification" status notes — a new pattern

T-012 introduced a previously-unseen pattern in the audit log: each new entry (UNSAFE-2026-0019 / 0020 / 0021) ships with an explicit "Pending QEMU smoke verification at the maintainer's first opportunity per the T-012 review-history row" status note. The note flags that:

- The kernel-build-clean half is verified in this dev environment.
- The actual MMIO discipline + IRQ-take + dispatch path is observable only in QEMU smoke.
- The maintainer lifts the note via append-only Amendment after running the smoke trace.

**Why this is new.** UNSAFE-2026-0001 .. UNSAFE-2026-0018 were all entered as `Status: Active.` without a verification-pending qualifier — either because their verification was kernel-build-only (e.g. `Send`/`Sync` markers) or because QEMU smoke had run pre-merge (e.g. UNSAFE-2026-0017's EL drop). T-012 is the first arc where:

- The pre-merge kernel-build is genuinely insufficient for verification (you can't test "the GIC actually delivers an IRQ" without QEMU smoke).
- The smoke run is genuinely deferred (the maintainer's environment, not the dev environment).
- The pattern of "ship the entry now; lift verification status later via Amendment" is the right shape rather than holding the entry until verification completes.

**The introducing-commit boundary (UNSAFE-2026-0017's "Discipline note for future readers" paragraph) means the *body* of each entry was locked at the introducing commit. Only the verification-status sub-claim is mutable, and only via append-only Amendment.** This is consistent with `unsafe-policy.md` §3's two-form-update rule (status change + Amendment); the status note here is a hybrid — it pre-declares the future status change so a reader knows what shape the Amendment will take.

**Lesson: explicit verification-pending status notes are worth canonising.** When an audit entry ships ahead of full verification, an explicit note inside the entry's `Status:` line — flagging *what* hasn't been verified, *who* will verify, and *what mechanism* (append-only Amendment vs. status flip vs. Removed) lifts it — is a strict improvement over the current `Status: Active.` shape. **Threshold for canonising as an `unsafe-policy.md` §3 sub-rule: a *second arc* needs to reproduce the pattern**, so the rule rests on cross-arc reproducibility rather than three same-arc entries. T-012's three new entries (UNSAFE-2026-0019 / 0020 / 0021) all share one arc and one shape; they are one data point in the cross-arc sense. The next IRQ-related arc (preemption / `time_sleep_until` syscall) that ships another deferred-verification entry is the second data point that triggers canonisation.

### ADR-0025 §Rule 1 + §Rule 2 — second empirical run

**Rule 1 (forward-reference contract): held.** ADR-0021's 2026-04-28 Amendment forward-referenced UNSAFE-2026-0014's Amendment (T-012 commit `28c5ce9`); both landed in the same commit. No ungrounded forward-reference rotted.

**Rule 2 (rider rate as signal): one ADR's first-week measurement still pending; B1 ADR activity was a single planned Amendment.**

- ADR-0024 (Accepted 2026-04-27) — 0 riders in first ~24 hours. The first-week measurement is scheduled for **2026-05-04** per the B0 retro's Adjustments item. T-013 + T-012 implementation arc did not produce any ADR-0024 rider; the rule's threshold is "3+ riders in a week" so 0 is a clean signal.
- ADR-0021 (Accepted 2026-04-22) — 1 Amendment in B1 (the 2026-04-28 IRQ-handler aliasing-discipline extension). This was forward-anticipated in the original ADR's *Decision outcome* and in T-012's task file from the moment T-012 was opened, so it's a *planned* Amendment, not a "rider rate" signal of post-Accept drift. **A planned Amendment that lands when its trigger fires is what a healthy ADR looks like; the rule should not treat it as a rider-rate signal.**

**Recommended rule clarification:** add a "*planned* vs. *unplanned* Amendment" distinction to ADR-0025 §Rule 2's audit-threshold counting. Planned Amendments (forward-referenced from the originating ADR's body, with a trigger condition that fires deterministically on a downstream task) should not count toward the "3+ in a week" threshold; unplanned Amendments (post-Accept reasoning corrections, scope drifts, body errata) should. Defer to the maintainer at the 2026-05-04 measurement; if ADR-0024's rider rate is also 0, the planned/unplanned distinction is hypothetical until a post-Accept rider actually appears.
