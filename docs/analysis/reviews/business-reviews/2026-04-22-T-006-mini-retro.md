## What we learned

**The `&mut self` trap is ergonomic, not technical.** The first-draft ADR assumed the hazard lived in the parameters, not the receiver. Method syntax is so thoroughly internalised that "of course the scheduler takes `&mut self`" survived the initial option-analysis pass; only the second read of the draft noticed that `SCHED.assume_init_mut()` at the BSP call site produces exactly the `&mut Scheduler` we claimed to have eliminated. **The lesson is not "always use raw pointers"** — it is that aliasing arguments must enumerate the call graph, including the caller's frame, rather than only reasoning about the called function's signature. Future ADRs touching aliasing (per-CPU locks in Phase C, preemption in later Phase B) should make "trace who holds what across the switch or the lock" an explicit checklist item.

**Design discipline survives if you enforce it on the boring entry points too.** `Scheduler::start` is called exactly once and its `&mut Scheduler` lives on a frame that is never resumed, so it was safe *in practice*. Leaving it as `&mut self` turned out to be worse than fixing it: the audit-log retirement now needs a rider explaining why the "all shared-state accessed as raw pointer" invariant has one exception. One exception invites another. The follow-up commit restores the clean statement.

**Peer review caught what self-review missed.** Today's second-pass review flagged three silent-failure paths that host tests would never cover: `let _ = self.ready.enqueue(...)` (task loss if capacity assumption breaks), `sync_generation` dropping an in-flight `Capability` on slot reuse (unreachable today, but reachable the moment Phase B adds endpoint destruction), and `ipc_notify` paired with no unblock path (fine today because no blocking wait API exists, but a trap for the next contributor who adds one). None of these are bugs on `development` today. All three are regressions-in-waiting. **Preventative `debug_assert` / `# Panics` / doc-comment work around invariants that "can't happen today" is cheap and worth doing in the same commit as the invariant.** The Phase A review workflow already catches this; T-006 confirms it catches ADR-driven refactors too.

**ADR revisions inline vs. follow-up ADR** — handling the `start()` fix as a rider on ADR-0021 rather than as ADR-0021a or a new ADR worked. The rule emerging: if the follow-up strengthens the same decision without changing the chosen option's identity, it is a revision; if it overrides the option, it is a new ADR that supersedes. T-006 stayed firmly on the "revision" side.
