## 1. Capability correctness

Adversarial frame: *can a caller invoke `cap_copy`, `cap_derive`, `cap_revoke`, `cap_drop`, `cap_take`, `ipc_send`, `ipc_recv`, or `ipc_notify` without holding the authority the security-model document requires?*

- **Every privileged table op requires a right bit.** OK — `cap_copy` rejects without `DUPLICATE` ([`kernel/src/cap/table.rs:185-187`](../../../../kernel/src/cap/table.rs#L185-L187)); `cap_derive` rejects without `DERIVE` ([`kernel/src/cap/table.rs:260-262`](../../../../kernel/src/cap/table.rs#L260-L262)); `cap_revoke` rejects without `REVOKE` ([`kernel/src/cap/table.rs:322-324`](../../../../kernel/src/cap/table.rs#L322-L324)). Rights are always checked before any slot mutation.
- **IPC rights are enforced strictly per kind.** OK — `validate_ep_cap` requires the supplied `CapRights` *and* matches the object variant, rejecting `Notification`/`Task` caps at the `ipc_send`/`ipc_recv` door ([`kernel/src/ipc/mod.rs:371-386`](../../../../kernel/src/ipc/mod.rs#L371-L386)). `ipc_notify` requires `NOTIFY` on a `Notification` object only ([`kernel/src/ipc/mod.rs:388-402`](../../../../kernel/src/ipc/mod.rs#L388-L402)).
- **Narrowest-sufficient rights.** OK — `SEND`, `RECV`, `NOTIFY`, `TRANSFER` are distinct one-bit rights, not grouped, so a `SEND`-only cap cannot `recv` and vice versa. The demo in `bsp-qemu-virt` grants `SEND | RECV | DUPLICATE` to both tasks ([`bsp-qemu-virt/src/main.rs:382`](../../../../bsp-qemu-virt/src/main.rs#L382)). See §8 for a follow-up note on the symmetric grant.
- **Check-before-side-effect on `cap_copy`.** OK — source rights and object are snapshotted, DUPLICATE/WidenedRights checked, and only then `pop_free` mutates the slot array ([`kernel/src/cap/table.rs:174-207`](../../../../kernel/src/cap/table.rs#L174-L207)). A failed check leaves free-list invariants intact.
- **Check-before-side-effect on `cap_derive`.** OK — same pattern; `pop_free` is only reached after DERIVE + subset + depth checks ([`kernel/src/cap/table.rs:255-279`](../../../../kernel/src/cap/table.rs#L255-L279)).
- **Check-before-side-effect on `ipc_send` queue-full pre-flight.** OK — `peek_state` is non-destructive; the `QueueFull` return leaves both endpoint state and the caller's cap table unchanged ([`kernel/src/ipc/mod.rs:247-254`](../../../../kernel/src/ipc/mod.rs#L247-L254)). The regression test `send_with_bad_transfer_cap_preserves_recv_waiting` captures the invariant.
- **Check-before-side-effect on `ipc_recv` cap install.** OK — `ReceiverTableFull` is detected via `is_full()` *before* `core::mem::replace` moves the pending state to `Idle`. Inverting the order would leak the in-flight capability ([`kernel/src/ipc/mod.rs:310-324`](../../../../kernel/src/ipc/mod.rs#L310-L324)). Worth restating: if `insert_root`'s error contract later broadens beyond "table full", this pre-check becomes insufficient — a note in-comment already flags this ([`kernel/src/ipc/mod.rs:312-316`](../../../../kernel/src/ipc/mod.rs#L312-L316)).
- **Capability transfer is move-only, enforced at the type level.** OK — `Capability` is `#[derive(Debug)]` but deliberately not `Copy`/`Clone` ([`kernel/src/cap/mod.rs:100-104`](../../../../kernel/src/cap/mod.rs#L100-L104)). Duplication is structurally impossible without going through `cap_copy`, and `cap_copy` requires `DUPLICATE`.
- **Transfer-over-IPC is atomic on the sender side.** OK — `cap_take` removes the capability from the sender's table and hands ownership to the endpoint's `SendPending`/`RecvComplete` state in a single uninterruptible kernel path ([`kernel/src/ipc/mod.rs:256-277`](../../../../kernel/src/ipc/mod.rs#L256-L277)). If `cap_take` fails (e.g. the transfer cap had children), endpoint state is untouched.
- **Transfer requires `TRANSFER` right on the transferred cap.** OK — pre-flight in `ipc_send` loads the transfer cap, checks it contains `TRANSFER`, and rejects with `InvalidTransferCap` otherwise ([`kernel/src/ipc/mod.rs:232-239`](../../../../kernel/src/ipc/mod.rs#L232-L239)). Regression test `send_without_transfer_right_on_xfer_cap_fails` in the IPC tests.
- **Revocation is atomic against readers inside the kernel path.** OK — `cap_revoke` builds the full BFS descendant list into a fixed-size scratch array and then frees all slots in one pass ([`kernel/src/cap/table.rs:337-390`](../../../../kernel/src/cap/table.rs#L337-L390)). Because the kernel is single-core and cooperative in v1, no other caller can observe a half-revoked tree. Multi-core revocation semantics are an explicit open question in `security-model.md` ("Revocation semantics under concurrent use").
- **Rights never broaden on derive/copy.** OK — both ops test `rights.contains(new_rights)` and return `WidenedRights` on violation ([`kernel/src/cap/table.rs:188-189, 263-264`](../../../../kernel/src/cap/table.rs#L188-L189)). Regression tests cover both.
- **Reserved rights bits are dropped at the ABI boundary.** OK — `CapRights::from_raw(bits)` silently masks `bits & KNOWN_BITS` so a hostile untrusted caller handing in `0x8000_0000 | something` cannot smuggle unknown flags past `contains` ([`kernel/src/cap/rights.rs:66-70`](../../../../kernel/src/cap/rights.rs#L66-L70)). A defense-in-depth choice that pays off when a future ADR adds a new bit and older callers are still in circulation.
- **Generation-tagged handles defeat slot-reuse confusion.** OK — every free bumps `generation` via `wrapping_add(1)` and the reuse test (`freed_slot_is_reused_with_bumped_generation`) proves stale handles fail lookup after reuse. **flagged — latent:** generation is a `u32`; `2**32` free-allocate cycles on one slot wraps it and resurrects a previously-invalidated handle. Not exploitable in A5/A6 (reuse count is single digits) but worth a tracking note before Phase B, because long-running services will churn capabilities continuously. Suggest an audit entry or ADR follow-up.
- **`cap_drop` cannot orphan descendants.** OK — interior nodes return `HasChildren`; only leaves can be dropped ([`kernel/src/cap/table.rs:418-430`](../../../../kernel/src/cap/table.rs#L418-L430)). Same for `cap_take`.
- **`ipc_recv` installs transferred caps as ROOTS in the receiver's table.** **flagged** — `install_cap_if_some` calls `table.insert_root(c)` ([`kernel/src/ipc/mod.rs:421-423`](../../../../kernel/src/ipc/mod.rs#L421-L423)). The derivation-tree relationship in the sender's table is discarded on transfer. Consequence: if Task A transfers a cap derived from a root Task A holds, and Task A later calls `cap_revoke` on that root, the copy now in Task B's table is **not** revoked — it is a root in B's table, unreachable from A's tree. The security-model invariant "Revocation is transitive over a derivation subtree" is satisfied **within a single table** but explicitly not across tables in v1. The security-model document frames this at the architectural level under *Authority transfer over IPC* and notes the parent-child relationship is "recorded for revocation" for `cap_derive`, but the v1 implementation restricts that recording to in-table derivation. This is consistent with seL4's pre-CDT scheme, but it means cross-table revocation is not yet supported. **Action requested:** either (a) document this limit explicitly in `security-model.md` Invariants / Open questions as a temporary divergence from the invariant, or (b) open a tracking task for a cross-table derivation tree in Phase B.
- **No silent capability cloning.** OK — the derived `Debug` on `Capability` is the only access path that could reveal a cap's innards, and `Debug` produces a format string, not a new `Capability` value. `CapRights` and `CapObject` are `Copy`, but the `Capability` wrapping them is not. Moving a cap through the table API is the only way to obtain another live handle.
- **A sender cannot transfer authority it does not hold.** OK — `cap_take` verifies handle validity + rights (via table lookup) before removing; the receiver only sees what was extracted.
- **`ipc_recv`'s "receiver table full" path does not drop the in-flight cap.** OK — the pre-flight check keeps the endpoint in its current state (`SendPending {cap: Some(_)}`), so a subsequent retry (after the receiver frees a slot) delivers the cap intact.

## 2. Trust boundaries

Adversarial frame: *in v1 the only boundary that exists in code is the kernel/BSP boundary and the internal Rust-API surface each task crosses when it calls into the kernel. There is no userspace/kernel privilege boundary yet (no syscalls, no EL0). What happens when untrusted bytes cross these interfaces?*

- **Userspace→kernel syscall boundary.** N/A — syscalls are Phase B; v1 is single-privilege EL1. `kernel_entry` and task entry functions are called from within the same privilege level and the same Rust module tree. The boundary exists *in the security-model document* but is not code yet. Re-examine at the first syscall.
- **`CapHandle` values crossing an (eventually external) boundary.** OK — `CapHandle` is an opaque (`index`, `generation`) pair. `index()` and `generation()` accessors are `#[doc(hidden)]` and used only by tests ([`kernel/src/cap/table.rs:47-59`](../../../../kernel/src/cap/table.rs#L47-L59)). A fabricated handle (wrong generation, or index past capacity) is rejected by `resolve_handle` which returns `InvalidHandle` ([`kernel/src/cap/table.rs:506-518`](../../../../kernel/src/cap/table.rs#L506-L518)). The index bounds-check uses `.get()` not `[index]`, so out-of-range indices return `None` rather than panicking.
- **`CapRights::from_raw` at the ABI boundary.** OK — covered in §1. Reserved-bit masking is in place today so Phase-B syscall bindings can trust incoming bits.
- **`Message` validation.** OK — `Message { label: u64, params: [u64; 3] }` is `#[derive(Copy, Clone, Debug, Default, Eq, PartialEq)]` plain data, passed by value ([`kernel/src/ipc/mod.rs:62-68`](../../../../kernel/src/ipc/mod.rs#L62-L68)). The kernel does not interpret message contents, and Phase A's in-kernel callers cannot construct invalid variants because there are no invalid variants. Bytes from "userspace" (which today are another in-kernel task) cannot trigger parser-state bugs because there is no parser.
- **Buffer length checks.** N/A — Phase A IPC has no buffer/pointer parameters. When `ipc_send` grows a buffer-carrying variant (Phase B: shared-memory IPC), re-review.
- **No kernel dereference of raw user pointers.** OK — grep confirms zero `*const u8` / `*mut u8` dereferences inside `kernel/src/` (the `stack_top` argument is forwarded to the BSP and written into an `Aarch64TaskContext` as an integer value; it is never dereferenced in-kernel). `ipc_recv_and_yield`'s call chain handles pointers only inside the BSP's `context_switch_asm`.
- **Cross-task IPC does not grant authority the sender did not hold.** OK — `cap_take` reads rights + object from the sender's own table entry; nothing is synthesised.
- **Stale endpoint slot reuse does not carry over waiter state across generations.** OK — `IpcQueues::sync_generation` resets the slot state to `Idle` when `slot_generations[idx] != handle.generation()` ([`kernel/src/ipc/mod.rs:181-189`](../../../../kernel/src/ipc/mod.rs#L181-L189)). Covered by the regression test `stale_queue_state_reset_on_slot_reuse`. Malicious reuse attempt: free an endpoint that was `RecvWaiting`, allocate a new one into the same slot, expect the new one to deliver to the old waiter — defeated by the generation match.
- **Boot information handed from BSP to kernel.** OK — today there is no info payload. `kernel_entry()` takes no arguments and `boot.s` does not forward the QEMU DTB pointer ([`bsp-qemu-virt/src/boot.s:19`](../../../../bsp-qemu-virt/src/boot.s#L19)). When DTB/bootinfo is introduced, the parser must validate every field.
- **The scheduler does not trust task-supplied register values.** OK — `init_context` writes `lr = entry as u64` (a function-pointer literal the kernel itself produces) and `sp = stack_top` (a static address produced by the BSP). Callees cannot influence these; there is no userspace pathway. When syscalls land and a task can request its own context layout, validate every register in that path.

## 3. Memory safety

Adversarial frame: *can a hostile caller, or a cooperative-scheduling-bug, cause any of the twelve audited unsafe regions to violate the invariants it claims?*

- **Kernel crate is zero-unsafe.** OK — `kernel/src/cap/`, `kernel/src/obj/`, `kernel/src/ipc/`, `kernel/src/sched/` each compile without any `unsafe` block of their own; every unsafe operation sits in the BSP or is forwarded via the `ContextSwitch` trait. `Grep` confirms only the sched-module `// SAFETY:` labels (nine occurrences) and they all wrap the BSP-provided `context_switch` / `init_context` calls.
- **UNSAFE-2026-0001 — `Pl011Uart::new(PL011_UART_BASE)` at kernel entry.** OK — MMIO base is a kernel-chosen constant; the QEMU virt window ownership is single-kernel in v1. Audit-log invariants match the SAFETY comment at [`bsp-qemu-virt/src/main.rs:346-348`](../../../../bsp-qemu-virt/src/main.rs#L346-L348). No uninitialized memory exposed — `Pl011Uart` is a single `base: usize`.
- **UNSAFE-2026-0002 — panic-handler Pl011Uart construction.** OK — per ADR-0007 `Console` is best-effort; interleaved writes are explicitly permitted. [`bsp-qemu-virt/src/main.rs:450-454`](../../../../bsp-qemu-virt/src/main.rs#L450-L454). Adversarial probe: a task could panic while the `Pl011Uart` is mid-transaction in another code path; result is garbled bytes, not memory corruption, because the hardware FIFO is the synchronization domain.
- **UNSAFE-2026-0003 / 0004 — `Send + Sync` on `Pl011Uart`.** OK — zero interior mutability; the only field is a `usize`. Marker impls, not pointer-safety claims. Audit entry invariants align with the in-code SAFETY ([`bsp-qemu-virt/src/console.rs:48-58`](../../../../bsp-qemu-virt/src/console.rs#L48-L58)).
- **UNSAFE-2026-0005 — MMIO volatile read/write in `write_bytes`.** OK — `read_volatile(base + 0x18)`, `write_volatile(base + 0x00)`. Both offsets are within the 4 KiB PL011 window; the base-address safety is inherited from `Pl011Uart::new`. Adversarial probe: `base.checked_add(UARTDR)` is not used — but `base` is a kernel constant (0x0900_0000), and both offsets are 0x18 and 0x00 which cannot overflow a `usize` given the realistic `base` value. On 32-bit targets this would warrant an add-with-overflow check, but Umbrix is aarch64-only.
- **UNSAFE-2026-0006 — `Send + Sync` on `QemuVirtCpu`.** OK — zero-size marker. DAIF and MPIDR are per-core registers, not shared memory. Multi-core re-review is flagged by the audit entry itself.
- **UNSAFE-2026-0007 — inline asm for DAIF/MPIDR/WFI/ISB.** OK — all five callsites use `options(nostack, nomem)` correctly: DAIF and MPIDR do not touch memory; WFI and ISB do not touch memory. `MRS DAIF` reads only; `MSR DAIFSET, #0xf` sets atomically. `restore_irq_state` writes the full DAIF value — its safety is a *caller contract* that the caller round-trips the `IrqState` unmodified. The `IrqGuard` RAII wrapper ([`hal/src/cpu.rs:107-122`](../../../../hal/src/cpu.rs#L107-L122)) is the only in-tree caller and satisfies the contract by construction.
- **UNSAFE-2026-0008 — context-switch assembly.** OK — `context_switch_asm` is `#[unsafe(naked)]` and uses `naked_asm!` as its sole body, so there is no compiler-generated prologue/epilogue to alter SP before the `mov x8, sp` save ([`bsp-qemu-virt/src/cpu.rs:189-240`](../../../../bsp-qemu-virt/src/cpu.rs#L189-L240)). Callee-saved set is complete per AAPCS64: `x19`–`x28`, `x29` (fp), `x30` (lr), `sp`, `d8`–`d15` (lower 64 bits of `v8`–`v15`). Adversarial audit of saved set: (i) `x30` saved and restored — OK; (ii) `sp` moved through `x8` scratch because `sp` cannot be a direct store operand — OK; (iii) `d8`–`d15` lower halves saved as 64-bit stores via `stp d8, d9, ...` — OK; (iv) callee-saved GPRs `x19`–`x28` — OK; (v) TPIDR_EL0/TPIDRRO_EL0 not saved — intentional, these are TLS registers and the v1 kernel does not use TLS; if Phase B introduces TLS, revise. **No deviation found between in-code SAFETY and UNSAFE-2026-0008 audit entry invariants.**
- **UNSAFE-2026-0009 — `init_context` writes entry pointer + stack.** OK — `ctx.lr = entry as usize as u64; ctx.sp = stack_top as u64;` ([`bsp-qemu-virt/src/cpu.rs:265-273`](../../../../bsp-qemu-virt/src/cpu.rs#L265-L273)). The `fn() -> !` type ensures the entry truly never returns; if it did, `ret` would jump to whatever's at the default-zero `lr` (address 0, likely trap). Rust's type system enforces the non-returning discipline. Stack alignment: `TaskStack` is `#[repr(C, align(16))]` and `top()` returns `ptr.add(4096)` which is 16-byte aligned because `4096 % 16 == 0` ([`bsp-qemu-virt/src/main.rs:94-122`](../../../../bsp-qemu-virt/src/main.rs#L94-L122)). 4 KiB is well above the 512-byte minimum.
- **UNSAFE-2026-0010 — `Sync` on `StaticCell<T>`.** OK — single-core cooperative invariant holds the "only one task at a time" property. Once preemption or SMP lands, this marker is wrong and must be replaced. Audit entry explicitly flags this.
- **UNSAFE-2026-0011 — `Sync` on `TaskStack`.** OK — same reasoning; each stack is used by exactly one task; the `UnsafeCell<[u8; 4096]>` interior is never materialised as a safe reference.
- **UNSAFE-2026-0012 — `&mut` aliasing across cooperative yields.** **flagged — known, documented, not resolved** — this is the most significant memory-safety concern in the tree. The `task_a` and `task_b` functions hold `&mut SCHED`, `&mut EP_ARENA`, `&mut IPC_QUEUES`, `&mut TABLE_{A,B}` live across `cpu.context_switch`; when the other task resumes, it derives fresh `&mut` references to the first three of those. Rust's aliasing model says two live `&mut` to the same `UnsafeCell` interior is immediately UB regardless of whether they are accessed simultaneously — the compiler is entitled to treat one of them as unaliased for optimisation purposes. The audit entry's invariants list the operational mitigations (single-core cooperative; no access across the switch point), and the QEMU smoke test passes, but the language-level contract is violated. **The audit entry status is `Active — to be resolved by raw-pointer API refactor (future ADR)`**, which correctly reflects the situation. Security-review verdict on this specific item: documented, mitigated operationally, not actually safe under strict aliasing. **Blocker for Phase B.** A raw-pointer refactor or a redesign of the task-entry API that does not hold `&mut` across the switch is required before preemption / SMP can land, and should be the first item on Phase B.
- **`# Safety` sections hold for every caller.** OK — `ContextSwitch::context_switch` and `::init_context` ([`hal/src/context_switch.rs:40-70`](../../../../hal/src/context_switch.rs#L40-L70)) specify their invariants; both in-tree callers (`Scheduler::start`, `Scheduler::yield_now`, `Scheduler::ipc_recv_and_yield`, `Scheduler::add_task`) mask interrupts via `IrqGuard` and supply initialised contexts. No third caller exists.
- **No uninitialized memory exposed.** OK — `Aarch64TaskContext` derives `Default`, all fields are `u64`, so `init_context` writes into a zero-initialised frame and the remaining fields are provably zero. First restore pops zeros into `x19`–`x28`/`fp`/`d8`–`d15`, which is safe because the entry function (a `fn() -> !`) re-establishes its own frame.
- **`MaybeUninit::assume_init_ref`/`assume_init_mut` sites.** OK — each is guarded by a `SAFETY:` comment and all are after the single-write-at-kernel_entry window. Adversarial audit: if `kernel_entry` panicked between the `write(...)` and `start(...)` calls, a subsequent `assume_init_ref` would be UB. But the panic handler is `!` (spins forever), so no task runs after such a panic. OK.
- **Use-after-free on capability slots.** OK — slot reuse bumps generation; stale handle fails lookup (see §1 generation-wrap flag).
- **Use-after-free on kernel-object slots.** OK — same pattern in `Arena`.
- **Scheduler context-array aliasing via split-borrow.** OK — `let ctx_ptr = self.contexts.as_mut_ptr(); let cur = &mut *ctx_ptr.add(current_idx); let nxt = &*ctx_ptr.add(next_idx);` ([`kernel/src/sched/mod.rs:310-316`](../../../../kernel/src/sched/mod.rs#L310-L316), [`:406-410`](../../../../kernel/src/sched/mod.rs#L406-L410)). The SAFETY comment asserts `current_idx != next_idx`. Audit probe: can the two indices ever be equal? In `yield_now` the `Some(h) if h != current_handle` guard prevents it ([`kernel/src/sched/mod.rs:289-297`](../../../../kernel/src/sched/mod.rs#L289-L297)); in `ipc_recv_and_yield` `self.current` is set to `None` before dequeuing `next_handle`, and `next_idx` is derived from a fresh dequeue — but `current_idx` was computed *before* clearing `self.current`, so the invariant is that the currently-blocking task is not in the ready queue. That holds because only the currently-running task is *not* in the queue; every other ready task is. If, hypothetically, the same task handle appeared twice in the queue, the second dequeue could equal `current_handle` — but the scheduler never enqueues a running task twice (it dequeues before running, re-enqueues on yield). OK.

## 4. Kernel-mode discipline

Adversarial frame: *can a call from a task wedge the kernel, force a kernel panic, or starve another task's work?*

- **No allocation in ISR.** OK — no ISRs exist in v1 (no interrupt sources configured). When timer / IRQ controller is enabled in Phase B, re-audit.
- **Bounded kernel state.** OK — every kernel data structure is compile-time-bounded: `CAP_TABLE_CAPACITY=64` per table, `TASK_ARENA_CAPACITY=16`, `ENDPOINT_ARENA_CAPACITY=16`, `NOTIFICATION_ARENA_CAPACITY=16`, `MAX_DERIVATION_DEPTH=16`, `SchedQueue<N>` capacity pinned to `TASK_ARENA_CAPACITY`. This satisfies the `security-model.md` "Bounded kernel resources" invariant. Every over-capacity path returns a typed error (`CapsExhausted`, `ArenaFull`, `QueueFull`, `DerivationTooDeep`, `HasChildren`).
- **Allocation paths return typed errors, not panics.** OK in the cap/obj/ipc subsystems — every over-capacity path is a typed error. **flagged — `Scheduler::start` panics on empty ready queue** ([`kernel/src/sched/mod.rs:246-253`](../../../../kernel/src/sched/mod.rs#L246-L253)). The allow-attribute calls this a "kernel programming error", which is true in v1 (the BSP always adds two tasks), but turning a configuration bug into a panic in the scheduler's boot path is unnecessary: the caller supplied data, the scheduler could return `SchedError::QueueEmpty` and let the BSP choose to halt. Recommend refactoring to a typed error in Phase B. Low severity today because the call site is a single known-good path in the BSP; gets higher severity when kernel-bringup code is more dynamic.
- **Allocation paths return typed errors — `Scheduler::ipc_recv_and_yield` deadlock panic.** **flagged** — when every ready task is blocked on IPC, `self.ready.dequeue()` returns `None` and the function `panic!("deadlock: …")` ([`kernel/src/sched/mod.rs:388-395`](../../../../kernel/src/sched/mod.rs#L388-L395)). A malicious task (Phase B+ threat) could potentially arrange the endpoint topology so every task is waiting on a peer. In Phase A this is a kernel-bug trap; in Phase B it is an attack surface because capability-authorised IPC is how userspace talks to the kernel. The ADR-0019 open-questions list already includes "resource-exhaustion DoS policy within the scheduler" and an idle task; this panic must be eliminated before userspace exists. Recommend: idle task in Phase B, return `Err(SchedError::Deadlock)` rather than panic.
- **Unbounded loops.** OK — `cap_revoke`'s BFS is bounded by `CAP_TABLE_CAPACITY` with an explicit `desc_len < CAPACITY` check ([`kernel/src/cap/table.rs:345-385`](../../../../kernel/src/cap/table.rs#L345-L385)). `unlink_from_siblings` walks a sibling chain bounded by the same capacity. `unblock_receiver_on` iterates `0..TASK_ARENA_CAPACITY`. `SchedQueue`'s operations are O(1). `write_bytes` spins on `UARTFR_TXFF` without a bound — acceptable for a best-effort diagnostic console, but a broken/absent UART would hang the kernel. Phase B could add a timeout or configurable attempt count; low priority.
- **Critical sections minimised.** OK — `IrqGuard` is held only across the `cpu.context_switch` call in `yield_now`, `start`, and `ipc_recv_and_yield`. The rest of each function runs with interrupts enabled-or-as-caller-state.
- **No new kernel panics on hot paths — clippy enforcement.** OK — the kernel crate has `#![deny(clippy::panic)]` and `#![deny(clippy::unwrap_used)]` ([`kernel/src/lib.rs:38-40`](../../../../kernel/src/lib.rs#L38-L40)). Both explicit panics in `sched/mod.rs` carry `#[allow(clippy::panic)]` attributes with reasons; they are isolated and do not appear elsewhere. **flagged — `debug_assert!` in `ipc_recv_and_yield` resume path** ([`kernel/src/sched/mod.rs:417-421`](../../../../kernel/src/sched/mod.rs#L417-L421)) compiles out in release, leaving the silent Pending-as-Ok fallthrough if a scheduler invariant ever broke. Recommend: convert to a hard return `Err(SchedError::Ipc(IpcError::InvalidCapability))` or a new error variant, so release builds cannot silently propagate a broken invariant to the caller.
- **Scheduler `yield_now` single-task path.** OK — if the queue is empty after re-enqueuing current, the function returns without switching; `self.current` is unchanged. No deadlock, no panic, fair semantics ([`kernel/src/sched/mod.rs:289-297`](../../../../kernel/src/sched/mod.rs#L289-L297)).
- **BSP `task_a`/`task_b` `.expect(...)` on IPC failures.** **flagged — BSP discipline, not kernel discipline** — BSP task-body code calls `.expect("task A: ipc_send failed")` ([`bsp-qemu-virt/src/main.rs:285, 303`](../../../../bsp-qemu-virt/src/main.rs#L285)) and `panic!("task B: expected Received outcome …")` ([`bsp-qemu-virt/src/main.rs:201, 308`](../../../../bsp-qemu-virt/src/main.rs#L201)). The BSP crate does not inherit the kernel crate's `#![deny(clippy::panic)]`. This is acceptable for a demo: a panic in a task body is the Phase-A equivalent of fault containment (the kernel itself survives since the panic handler halts only that task's execution on its own stack — but with no supervisor, the whole kernel spin-loops). Before any real driver is written as a task, the project needs a supervisor endpoint design and `error-handling.md § 8` fault containment wired up. Not a Phase-A blocker; **pre-requisite for Phase B's first driver task**.
- **Arithmetic-side-effect lint coverage.** OK — `#![deny(clippy::arithmetic_side_effects)]` in the kernel crate forces explicit `saturating_add` / `wrapping_add` with reason-annotated `#[allow]` where used. `SchedQueue` uses `wrapping_add` and `%` with an explicit bound proof; `Arena::allocate`, `CapabilityTable::pop_free`, and the cap-revoke BFS all use bounded saturating arithmetic.
- **Float arithmetic ban.** OK — `#![deny(clippy::float_arithmetic)]` in the kernel lib. The d8–d15 register saves are integer `STP d8, d9` instructions; no floating-point *operations* are performed, only register preservation.

## 5. Cryptography

**N/A — Umbrix has no cryptographic primitives in any of the committed code.** `Grep` confirms zero matches for `cipher|hash|signature|kdf|nonce|sha|aes|curve|hmac|rand` inside `kernel/`, `hal/`, and `bsp-qemu-virt/`. The `security-model.md` document is explicit that crypto lands in later phases, one ADR per primitive, and must use audited crates not roll-your-own. Revisit at first primitive introduction.

## 6. Secrets and logging

Adversarial frame: *what does a task's output (console, panic message) reveal about kernel-internal state?*

- **Capability bits never appear in log output.** OK — `Capability` derives `Debug` but is never written to the console. `CapHandle` is `Debug`, and its accessors are `#[doc(hidden)]`, but no log line in the BSP formats a `CapHandle`. `Grep` in the BSP for `CapHandle|Capability|CapObject` inside `writeln!`/`write_bytes`/`FmtWriter` argument lists: no hits.
- **Task labels are user-defined.** OK — the demo uses `0xAAAA`/`0xBBBB` and prints them, which is fine (not a secret).
- **Panic-info format may leak.** OK — the panic handler prints `{info}`, which for Rust panics includes the source location and the custom message. Panic messages in the kernel are carefully worded ("deadlock: all tasks blocked on IPC and no idle task available", "Scheduler::start called with empty ready queue") and do not carry capability bits or task identifiers that would help an attacker target a specific task. If future panics include task state, review them.
- **`Debug` impls on security-sensitive types.** OK — `Capability` derives `Debug`; the derived impl prints `rights` (bitfield integer) and `object` (a typed handle — index + generation). In v1 there is no adversary to observe these; in Phase B, these derived `Debug` impls must be replaced or redacted if `Capability` becomes an argument to any logging primitive exposed to userspace. Not a v1 blocker but a standing rule.
- **`Notification::word` visibility.** OK — notifications store a single `u64` word; the v1 demo does not print it, and Phase A has no "dump all notifications" path.
- **MMIO base addresses in panic output.** OK — no code paths include `PL011_UART_BASE` in a diagnostic message. On a future MMU-bound system this would be meaningless to an attacker anyway (physical address).
- **Uninitialised stack exposure.** OK — task stacks are `[0u8; 4096]` (zero-initialised at the definition site via `const fn`). The `#[repr(C, align(16))]` `TaskStack` does not contain any field that could leak.
- **Task `Debug` impl.** OK — `Task { id: u32 }` is `Debug`; `id` is not secret. Same for `Endpoint { id: u32 }` and `Notification { word: u64 }` (noting §6 above on Notification's word becoming sensitive once userspace can set bits representing private events).

## 8. Threat-model impact

Adversarial frame: *does any code path in v1 change what the system defends against vs. the statements in `security-model.md`?*

- **Capability-based microkernel shape.** OK — v1 implements the capability table, rights, derivation tree, revocation, transfer, and the `SEND`/`RECV`/`NOTIFY`/`TRANSFER`/`DUPLICATE`/`DERIVE`/`REVOKE` initial right set. This matches the "Capabilities" section of `security-model.md` verbatim at the level v1 can demonstrate; `MemoryRegion` is correctly deferred to Phase B (noted in `kernel/src/cap/mod.rs:58-60`). No divergence.
- **"No privileged operation without authorising capability" invariant.** OK — every `ipc_send` / `ipc_recv` / `ipc_notify` / `cap_copy` / `cap_derive` / `cap_revoke` rejects on missing rights before any side effect. Scheduler `yield_now` / `start` are kernel-internal (no capability is defined for them in v1); these will need a `TaskCap` `Kill` right or similar when tasks can control peers. Not a v1 divergence.
- **"Capabilities move with consent" invariant.** OK — `cap_take` consumes the sender's handle; IPC transfers materialise in the receiver only after the move succeeds.
- **"Capability rights narrow, never broaden, on derivation" invariant.** OK — enforced structurally by `rights.contains(new_rights)`.
- **"Revocation is transitive over a derivation subtree" invariant.** **flagged — partial divergence** — revocation is transitive *within a single table*, but IPC transfer creates a new root in the receiver's table, discarding the cross-table parent-child link (see §1). Either `security-model.md` needs an explicit clause — "derivation-tree revocation is single-table in v1; cross-table CDT is Phase B work" — or Phase B must implement a cross-table capability-derivation-tree. **Propose opening a tracking task:** *Cross-table capability derivation tree for cross-task revocation.*
- **"A sender cannot transfer authority it does not hold" invariant.** OK — `cap_take` validates; `TRANSFER` right required.
- **"The kernel never dereferences raw userspace pointers" invariant.** OK today — no userspace pointers exist. Becomes live at Phase B.
- **"Drivers are userspace tasks" invariant.** OK — PL011 driver is in the BSP and runs at kernel privilege in v1 because there is no userspace. The design intent is userspace drivers; Phase B splits the PL011 out. Not a v1 divergence; a note.
- **"`unsafe` is audited" invariant.** OK — 12 entries, all referenced from corresponding SAFETY comments. No undocumented `unsafe` in the tree (enforced by `#![deny(clippy::undocumented_unsafe_blocks)]` workspace-wide).
- **"No proprietary binary blobs" invariant.** OK — build has no binary blobs; firmware is QEMU's built-in device model (not shipped by this project).
- **"Fault containment does not leak authority" invariant.** N/A (no faults yet — no exception handlers, no supervisors). Phase B.
- **"Bounded kernel state" invariant.** OK — covered in §4.
- **"DMA is capability-scoped where hardware permits" invariant.** N/A — v1 does not enable DMA-capable peripherals. QEMU virt exposes SMMUv3 but the BSP does not configure it; PL011 is PIO. Re-examine at the first DMA-capable driver.
- **Threat-model document completeness.** OK — `docs/architecture/security-model.md` exists and is thorough (it enumerates adversaries in scope, adversaries out of scope, trust boundaries, capability types, capability operations, memory-safety mechanisms, fault containment, bounded resources, side channels, cryptography posture, supply chain, and boot integrity). **Corrective item from §1:** the cross-table revocation limitation is not currently called out in the `Open questions` list — it should be, or a dedicated invariant-qualification note.
- **Two tasks each holding `SEND | RECV` on the same endpoint.** **flagged — operational, not architectural** — the A6 demo grants symmetric rights to both tasks ([`bsp-qemu-virt/src/main.rs:382`](../../../../bsp-qemu-virt/src/main.rs#L382)). With symmetric rights, either task could `ipc_recv` its own reply back, or send to itself. Correct delivery depends on the scheduler order (B added first, so B is in `RecvWaiting` before A sends). The ADR-0018 badge scheme (deferred) would let a server distinguish senders; without badges and without per-role asymmetric rights, there is no way for B to know the sender is A. For a two-task demo this is acceptable; for any system with more than two tasks it is insufficient. **Action requested:** re-state this limitation in `security-model.md`'s "Badges" paragraph (already present) *and* in the demo guide, so it is clearly a demonstration, not a recommended pattern.
- **Boot chain trust.** OK per architecture — pre-kernel trust implicit until measured boot lands. `boot.s` is minimal and auditable in-tree: stack pointer set, CPACR_EL1.FPEN enabled, BSS zeroed with 8-byte stores, branch to `kernel_entry`. DTB pointer is ignored. No EL transition. **Minor note:** `boot.s` does not explicitly mask interrupts (DAIFSet #0xf) before zeroing BSS. In v1 this is benign because QEMU drops the kernel into EL1 with interrupts effectively masked (DAIF is indeterminate at reset but no IRQ source is configured) and no timer or GIC is enabled. Hardening: add `msr daifset, #0xf` early in `_start` before the stack setup, so future BSPs that inherit this pattern are safe-by-default even if the reset state isn't. Not a blocker.
- **Secure / measured boot.** N/A — out of scope per `security-model.md`, deferred until deployment hardware supports it.
- **Side-channel posture.** OK — `security-model.md §Side channels` documents the current stance; v1 has no cryptography and no user/kernel privilege boundary, so side-channel surface is minimal today.

## Verdict

**Changes requested.**

Blocking items (must be addressed before Phase B begins or before the first userspace-facing entry point lands, whichever comes first):

1. **§1 / §3 — UNSAFE-2026-0012 (`&mut` aliasing across cooperative yields).** Formally UB under Rust's strict-aliasing rules; operationally safe under the v1 single-core cooperative invariant but structurally wrong. The audit entry already names a "raw-pointer API refactor" as the fix. Open a named ADR and a Phase-B task for the refactor; do it **before** preemption or SMP.
2. **§1 / §8 — cross-table capability revocation gap.** `install_cap_if_some` → `insert_root` discards the derivation parent on IPC transfer, so `cap_revoke` in the sender's table does not invalidate the copy in the receiver's table. Either update `security-model.md` to carve out v1's scope and add this to Open questions, or add a cross-table CDT to Phase B. The current gap silently violates the "Revocation is transitive over a derivation subtree" invariant as written.
3. **§4 — `Scheduler::ipc_recv_and_yield` deadlock panic.** Must become a typed error (or be eliminated by an idle task) before userspace tasks can reach it. A malicious Phase-B task that arranges an all-blocked topology should receive an error, not crash the kernel.

Non-blocking follow-ups (recommended; do not gate Phase A exit):

- **§1 — generation wrap-around.** `u32` generation wraps at 2^32 free-reuse cycles on the same slot. Not reachable in v1. Consider a successor ADR (larger generation, monotonic counter, or explicit handle-lifetime management) before a long-running service ships.
- **§4 — `Scheduler::start` panic on empty ready queue.** Convert to typed error.
- **§4 — `debug_assert!` in `ipc_recv_and_yield` resume path.** Convert to a typed error return so release builds cannot silently propagate a broken invariant.
- **§4 — BSP task-body `.expect`/`panic!`.** Needs a supervisor-endpoint design (fault containment) before the first real driver task in Phase B.
- **§4 — `write_bytes` unbounded spin on `UARTFR_TXFF`.** Add attempt-count cap for non-QEMU BSPs.
- **§8 — boot.s explicit `msr daifset, #0xf` early.** Defense-in-depth for future BSPs.
- **§8 — `security-model.md` cross-table revocation clause.** Add to Open questions or Invariants-with-scope.
- **§8 — demo-pattern warning.** Document that symmetric `SEND | RECV` grant on the same endpoint is a two-task-demo convenience, not a recommended topology; cite the badge-scheme (ADR-0018) deferral.
- **§6 — `Debug` on `Capability`.** Redact before the first user-observable log path ships.
- **§3 — cross-reference the UNSAFE-2026-0008 audit entry with AAPCS64.** The invariant list is correct today; if Phase B adds TLS (TPIDR_EL0), update the audit entry and the `context_switch_asm` save set in the same commit.
- **§7 — `cargo-vet init` now** so the baseline is empty-but-configured when the first dependency arrives.

**Follow-up tasks to open** (via `start-task`):

- *Eliminate UNSAFE-2026-0012 — raw-pointer scheduler API refactor* (blocks preemption / SMP).
- *Cross-table capability derivation tree for cross-task revocation* (blocks multi-task-system security-model compliance).
- *Scheduler-panic elimination — idle task and typed deadlock error* (blocks userspace exposure).

No axis was escalated beyond the review's scope; the three blockers are actionable inside the existing roadmap (Phase B) and do not require rewriting the threat model — only completing what it already promises.
