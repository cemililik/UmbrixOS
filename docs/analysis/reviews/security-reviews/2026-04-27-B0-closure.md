## 1. Capability correctness

Adversarial frame: *can a caller invoke `cap_copy`, `cap_derive`, `cap_revoke`, `cap_drop`, `cap_take`, `ipc_send`, `ipc_recv`, `ipc_notify`, `ipc_send_and_yield`, or `ipc_recv_and_yield` without holding the authority the security-model document requires?*

- **Phase A invariants intact.** All "OK" findings from the 2026-04-21 review (rights checked before mutation; check-before-side-effect; move-only Capability; `from_raw` reserved-bit masking) re-verified against the post-B0 source. No regressions.
- **`ipc_send_and_yield` / `ipc_recv_and_yield` rights enforcement.** OK — both bridge wrappers call `Scheduler::resolve_ep_cap` ([`kernel/src/sched/mod.rs:273-284`](../../../kernel/src/sched/mod.rs)), which performs the same `validate_ep_cap`-style lookup before any scheduler mutation. Failure path returns `SchedError::Ipc(InvalidCapability)` *before* any momentary `&mut Scheduler` is acquired. Regression test: T-011's `ipc_send_and_yield_send_error_preserves_scheduler_state` (kernel/src/sched/mod.rs:1492+) confirms the scheduler is untouched on the rights-fail path.
- **Capability transfer atomicity through the bridge.** OK — `ipc_send_and_yield`'s Phase 1 block calls `ipc_send` which goes through the same `take_cap_if_some`/`install_cap_if_some` pre-flight as the bare `ipc_send`. The bridge does not introduce a new cap-handling path; it only adds yield/unblock around the existing flow. The capability moves out of the sender's table inside the same momentary-`&mut` block that performs the endpoint-state transition.
- **`ReceiverTableFull` pre-flight cap retention.** OK — T-011's `recv_with_full_table_preserves_pending_cap` (kernel/src/ipc/mod.rs:1003+) is the regression test for the "in-flight cap stays parked in `SendPending` if the receiver's table is full" invariant the Phase A review flagged. The pre-flight check (`pending_has_cap && caller_table.is_full()`) is at `kernel/src/ipc/mod.rs:356-363`; nothing in B0 weakened it.
- **Slot-reuse generation guard with `Some(cap)` leak detection.** OK — `IpcQueues::reset_if_stale_generation` (`kernel/src/ipc/mod.rs:211-228`) fires a debug-mode assertion if a stale-generation reset would silently drop an in-flight `Capability`. T-011 added the paired tests (`stale_send_pending_with_some_cap_panics_in_debug` + `stale_recv_waiting_resets_silently` + `stale_send_pending_without_cap_resets_silently`) so the assertion's predicate is verified neither over- nor under-broad.
- **Cross-table revocation gap re-examined.** Unchanged from the Phase A review — `install_cap_if_some` still calls `insert_root` so transferred caps land as roots in the receiver's table, breaking transitive revocation across tables. The Phase A review action item (document explicitly OR open a tracking task) is now tracked as ADR-0023 (deferred per B0 closure plan); no new code makes the gap worse, no new code makes it better. Re-examine when the first multi-task server arc starts (B3–B6 timeframe).
- **Generation overflow re-examined.** Unchanged. Per the Phase A review's "flagged — latent": `u32` generations wrap at 2³² free-allocate cycles. B0 added no new generation-bumping paths beyond the existing `cap_drop`/`cap_take` flow. Long-running services concern still applies; tracking note rolls forward to B-late phases.
- **`SchedError::Deadlock` rollback scope.** OK — the typed return restores `s.current` and `s.task_states[current_idx]` to their pre-`ipc_recv_and_yield` values before returning Err (`kernel/src/sched/mod.rs:739-744`). The endpoint-state move from `Idle` to `RecvWaiting` is *not* rolled back (documented in `SchedError::Deadlock`'s doc comment); a subsequent retry sees `QueueFull`, not `Pending`. ADR-0022 records this as a deliberate v1 design call. Adversarial probe: a hostile task that triggers Deadlock then retries does not gain authority — the worst case is its own subsequent recv on the same endpoint sees `QueueFull` and must destroy/recreate the endpoint to recover. No capability leak; no escalation.
- **`PendingAfterResume` typed return.** OK — `ipc_recv_and_yield`'s resume-path (`kernel/src/sched/mod.rs:794-798`) decodes a `Ok(Pending)` from the resume-time `ipc_recv` into `Err(SchedError::Ipc(IpcError::PendingAfterResume))` instead of letting it propagate as `Ok(Pending)` (which a caller's `let RecvOutcome::Received { ... } else { panic!() }` would convert into a downstream panic). The typed return is the loud signal; the `debug_assert!` originally drafted to fire alongside was deliberately dropped (untestable in release-mode contexts that exercise the typed path). T-007's `ipc_recv_and_yield_resume_pending_returns_typed_err` test pins the contract.
- **`start_prelude` extraction does not bypass any check.** OK — the function's body is a copy of the original inner block in `start`, with no call sites elided and no rights checks removed. The only behavioural difference is testability (host tests can exercise the dequeue + state-mutation half without invoking the assembly context-switch). Audit cross-reference: UNSAFE-2026-0014's PR #9 Amendment correctly names `start_prelude` as a new site falling under the existing safety argument.

## 2. Trust boundaries

Adversarial frame: *what new boundaries did B0 add, and what untrusted bytes (or values) cross them?*

- **No userspace/kernel boundary yet.** Same as Phase A — syscalls are Phase B5+ work; the B0 arc did not introduce any new in-code boundary. `kernel_entry` and task entry functions are still called from within the same privilege level + same Rust module tree.
- **EL2 → EL1 boot transition (T-013).** OK — `boot.s`'s EL drop sequence configures `HCR_EL2 = (1 << 31)` (RW=1; everything else zero, explicitly non-VHE), `SPSR_EL2 = 0x3c5` (mode=EL1h + DAIF masked), `ELR_EL2 = post_eret`. The post-`eret` PSTATE inherits the masked DAIF from `SPSR_EL2`, so the K3-12 mask carries across the transition without a second `daifset`. UNSAFE-2026-0017's Invariants list documents the literal HCR_EL2 write rationale (RMW would preserve firmware's E2H/TGE/IMO/FMO/AMO at undesired values; ARMv8.0/8.1+ has no RES1 bits). Adversarial probe: a firmware that delivers at EL2 with `HCR_EL2.IMO = 1` (route IRQs to EL2) before `boot.s` writes HCR_EL2 — the literal write clears IMO so the kernel runs with EL1-bound IRQ delivery. Verified safe.
- **EL3 entry path.** OK — `boot.s` halts in a named-label `wfe`-loop on EL3 entry (`halt_unsupported_el: wfe ; b halt_unsupported_el`). No Rust runtime is up; no panic frame attempted. Adversarial frame: a future BSP for hardware that boots at EL3 would halt visibly rather than silently mis-execute. ADR-0024 §Open questions tracks the EL3→EL2→EL1 chain as a future BSP task.
- **`tyrne_hal::cpu::current_el()` cfg-gating.** OK — UNSAFE-2026-0018's Amendment correctly states the function is `cfg(all(target_arch = "aarch64", target_os = "none"))`-gated. On hosted targets (including `aarch64-apple-darwin`), the function is structurally absent (compile error) — there is no fallback path that runs the MRS at EL0 and yields `EL0`. Test mocks must declare an EL of their own; no helper-based escape hatch exists. Re-verified: `grep -r "current_el" --include="*.rs" .` in non-aarch64 contexts returns no matches.
- **`UNSAFE-2026-0016` post-condition contract.** OK — `QemuVirtCpu::new`'s assertion `current_el == 1` runs *before* any timer-related MRS (`CNTFRQ_EL0`, `CNTVCT_EL0`). The assertion's load-bearing-post-condition role (per UNSAFE-2026-0016's T-013 Amendment) means a regression in `boot.s`'s EL drop becomes a loud, named boot panic rather than silent timer misbehaviour. The assertion's panic message includes the observed EL, useful for triage.
- **GICv2 MMIO surface (deferred to T-012).** N/A — B0 did not introduce GIC MMIO. T-012 will, and a separate security review on T-012's tip is the right venue. The B1 closure security review or a T-012 dedicated review will cover that surface.
- **Bidirectional cross-references between ADRs and architecture docs.** N/A from a security perspective — these are documentation artefacts; no code or trust surface is altered. Worth noting as a *positive* for security reasoning: a future reviewer following an ADR's §References can now reach the architecture-doc summary that synthesises related ADRs without needing to know the cross-references implicitly.

## 3. Memory safety

Adversarial frame: *can a hostile caller, or a cooperative-scheduling-bug, cause any of the eighteen audited unsafe regions to violate the invariants it claims?*

- **Kernel crate is still zero-unsafe.** OK — `grep -r "unsafe" kernel/src/` returns only `unsafe fn` declarations in the raw-pointer scheduler bridge (which take `*mut` parameters and document the contract) and `// SAFETY:` comments on caller sites of those functions. No `unsafe { ... }` blocks inside the kernel crate's logic.
- **UNSAFE-2026-0012 (Removed) — `&mut Scheduler` aliasing across context switch.** OK — entry stays Removed, body intact, post-review rider notes the `start` extension to the discipline. Replaced by 0013 + 0014.
- **UNSAFE-2026-0013 — `StaticCell::as_mut_ptr`.** OK — the helper returns `*mut T` via `self.0.get().cast::<T>()`; no `&mut` materialised at any point. Layout-compat invariant (`MaybeUninit<T>` and `T` share layout) is a stdlib guarantee. Adversarial probe: a caller that returns the pointer from an inner scope and dereferences after the scope ends — the audit entry explicitly flags this as the caller's contract; the BSP's `task_a` / `task_b` derive each pointer once per bridge call and do not return them across scope boundaries. Verified by Miri's Stacked Borrows check (143/143 clean).
- **UNSAFE-2026-0014 (Active, with PR #9 Amendment) — Scheduler free-function momentary `&mut` pattern.** OK — the Amendment correctly names `start_prelude` (T-011) as a new site whose body matches the existing pattern. Adversarial trace of `start_prelude`:
  - Inner block: `let s: &mut Scheduler<C> = unsafe { &mut *sched };`
  - Mutates `s.task_states[next_idx]` and `s.current` only inside the block.
  - Block ends; `s` drops; `next_idx` returned by value.
  - Caller (`start`) takes its own raw-pointer-arithmetic path through `(*sched).contexts` for the context-switch — no re-borrow of `*sched` as `&mut` while the throwaway-context switch runs.
  No `&mut Scheduler<C>` is alive when `cpu.context_switch` would observe it. Miri verifies this.
- **UNSAFE-2026-0015 (Active, two pre-existing Amendments) — generic-timer MRS.** OK — Amendments cover (a) the CNTPCT→CNTVCT register-family swap and (b) the audit-readability "casual reader" pointer. T-013 did not modify the original-body `CNTVCT_EL0` / `CNTFRQ_EL0` reasoning. The post-T-013 EL precondition reasoning was strengthened: `boot.s`'s EL drop sequence makes the `current_el == 1` assertion a load-bearing post-condition rather than a defensive guard. The bsp/cpu.rs SAFETY comments in PR #9 round 2 correctly cite UNSAFE-2026-0017 + 0018 alongside 0015.
- **UNSAFE-2026-0016 (Active, T-013 Amendment) — boot-time `CurrentEL` self-check.** OK — Amendment records two concurrent shifts: (a) the assertion's role is now a load-bearing post-condition (not a defensive guard against a missing transition), and (b) the inline-asm MRS at the original site has been replaced by a call to `tyrne_hal::cpu::current_el()`. Behaviour and panic message unchanged. The Amendment dates the change and cites the helper's audit (UNSAFE-2026-0018).
- **UNSAFE-2026-0017 (Active, two PR #9 Amendments) — `boot.s` reset-vector DAIF mask + EL2→EL1 transition.** OK — `boot.s` source matches the audit entry's Operation field (after the GAS-syntax-correction Amendment). Adversarial trace of the `eret`:
  - `HCR_EL2 = 0x80000000` writes RW=1 only; E2H/TGE/IMO/FMO/AMO all 0.
  - `SPSR_EL2 = 0x3c5`: mode=EL1h, DAIF=masked.
  - `ELR_EL2` set via `adrp + add :lo12: post_eret` — PC-relative, valid regardless of load address.
  - `eret` consumes SPSR_EL2 + ELR_EL2; PSTATE.DAIF inherits SPSR_EL2.DAIF; CPU lands at EL1h with DAIF still masked.
  - First instruction at `post_eret` is the existing `adrp x0, __stack_top`. No window between `eret` and the SP setup where an interrupt could fire (DAIF still masked).
  Verified safe. The "Discipline note for future readers" Amendment paragraph correctly codifies the introducing-commit boundary as the body-locking event for §3 purposes; this resolves the PR #9 internal asymmetry where V13/V14 went via Amendment but V7 (HCR_EL2 paragraph) initially went in-place.
- **UNSAFE-2026-0018 (Active, one PR #9 Amendment) — `tyrne_hal::cpu::current_el` safe wrapper.** OK — single inline-asm MRS read inside an `unsafe` block, masked + shifted into a `u8` in 0..=3. Outer signature is safe; the Amendment correctly tightens the cfg-gating prose ("undefined at EL0 + traps with `SIGILL` on hosted" replacing the looser "would trap or yield `EL0`"). Adversarial probe: a caller from a non-aarch64 context — the `cfg` gate prevents compilation; cannot be reached. A caller from EL0 on aarch64 (no such caller in v1; would be a future userspace path) — `MRS CurrentEL` traps; the trap is fielded by the future EL1 trap handler (Phase B+ work). Today, no caller reaches EL0; today, the `cfg` gate is the active line of defence.
- **Cap-table sweep tests' debug-only `#[should_panic]` behaviour.** OK — `stale_send_pending_with_some_cap_panics_in_debug` is `#[cfg(debug_assertions)]`-gated, so it does not run under release-mode test compilation. The release-mode behaviour (debug_assert! is a no-op; the leak silently lands) is acceptable per `ipc/mod.rs`'s comment block: this is an in-debug guard against a future destruction path that should drain caps before bumping generations. The release-mode cap leak would surface as a generation-bump test failure in production, which would itself trigger investigation.
- **Architecture docs do not introduce new unsafe surface.** OK — T-008 added `scheduler.md` and `ipc.md` plus updates; both are pure documentation. No code changes; no new SAFETY claims to validate.

## 4. Kernel-mode discipline

Adversarial frame: *does any new code path violate the principle that kernel-mode work be minimal, deterministic, and capability-checked?*

- **`start_prelude` is module-private, not pub.** OK — `unsafe fn start_prelude<C: ContextSwitch + Cpu>(sched: *mut Scheduler<C>) -> usize` is `unsafe fn` (caller takes responsibility) but not `pub`. The only call site is `start`. A future caller that wants to reuse the prelude must take responsibility for the same safety contract; the function's doc-comment specifies the contract.
- **No new heap allocation.** OK — B0 added zero `alloc::*` imports, zero `Box::new`, zero `Vec::new`. The kernel remains heap-free per ADR-0001.
- **Idle task body.** OK — currently `core::hint::spin_loop()` + `yield_now`. Will move to `wait_for_interrupt + yield_now` when T-012 lands the IRQ infrastructure (per ADR-0022 first-rider sub-rider, gated on T-012). Pre-T-012, idle's spin is the explicit design call; not a CPU-burn bug.
- **Boot path discipline.** OK — `boot.s` is now ~100 lines of disciplined assembly; every block is documented; no out-of-spec instructions. K3-12 (DAIF mask) is the literal first instruction at `_start`. ADR-0024's "always drop to EL1" and the BSP boot checklist's §1 + §1a give the boot path explicit invariants.
- **Architecture docs reduce reasoning surface.** Indirect security positive: `scheduler.md` and `ipc.md` mean a security reviewer touching the IPC bridge or scheduler in a future PR has a synthesised picture to read against, instead of having to traverse 4+ ADRs to reconstruct intent. Lower review friction → higher review coverage.

## 8. Threat-model impact

Adversarial frame: *did the B0 arc reshape what the system defends against, or what it concedes to attackers?* Two ADRs in the arc shift load-bearing assumptions in the threat model; two more produce only positive observations about the review surface itself.

**ADR-0024 promotes EL1-only execution from defense-in-depth to a structural property.** Before T-013, the kernel relied on the firmware/emulator delivering at EL1 plus T-009's `UNSAFE-2026-0016` runtime check as a last-line guard against an EL2-delivered boot — defense-in-depth, with the assertion's panic as the loud-failure mode. Post-T-013, `boot.s` actively drives EL1 regardless of entry EL (or halts on EL3); UNSAFE-2026-0016 is now the post-condition assertion of that transition, not its only defence. The threat-model implication: the kernel's "we run at EL1" assumption is now an enforced invariant rather than a hoped-for one. A boot stack that delivers at EL2 (TF-A → U-Boot → kernel; QEMU `-machine virtualization=on`) was previously a silent risk surface; B0 closes that risk.

**ADR-0021 reshapes the aliasing model the kernel proves correctness against.** Before T-006, `&mut Scheduler<C>` aliasing across cooperative context switches was the latent UB documented as `UNSAFE-2026-0012` — the kernel's correctness depended on the optimiser not noticing the aliasing. Post-T-006, the raw-pointer scheduler bridge replaces `&mut Scheduler<C>` with `*mut Scheduler<C>` plus a momentary-`&mut` discipline that Stacked Borrows (Miri) verifies on every host-test run. The threat-model implication: the kernel's memory-safety story has shifted from "we hope the compiler doesn't exploit our UB" to "we have a checked invariant the test gate enforces." This is a strict improvement in what the kernel can be defended against — both compiler-evolution risk and human-introduced regression are now caught by Miri.

**Threat-model items inherited from Phase A (unchanged severity).** Both items predate B0; B0 added no new code that worsens or improves them, and ADR-0023's deferred-acceptance path remains the planned closure for the cross-table case:

- **Cross-table revocation gap** — tracked as ADR-0023 (deferred); re-examine when the first multi-task server arc starts in B3–B6.
- **Generation overflow (`u32`)** — tracked as a long-running-service concern; re-examine in B-late phases when continuous capability churn becomes realistic.

**Positive observations about the review surface itself** (not threat-model shifts in the kernel; observations about how this review and future reviews will work):

- **Audit-log Amendment discipline is now codified.** The introducing-commit boundary (not the merge boundary) locks an entry's body. UNSAFE-2026-0017's "Discipline note for future readers" paragraph is the canonical reference. Future security reviews should read this rule explicitly when checking entry-body integrity.
- **Architecture docs are a security multiplier.** `scheduler.md` and `ipc.md` mean the next security review can start from a synthesised intent statement rather than reconstructing intent from ADRs and source. Recommend that B1 / B2 architecture-doc work continue this pattern — `exceptions.md` for T-012's IRQ infrastructure would pay for itself in the next consolidated review.

## Verdict

**Approve.** All eight axes pass; no Yüksek findings; the two threat-model shifts B0 introduces (EL1 as structural invariant; raw-pointer aliasing model with Miri-checked Stacked-Borrows compliance) are net improvements to what the kernel can be defended against. Previously-flagged Phase-A inheritance items (cross-table revocation, `u32` generation overflow) are tracked at original severity with no B0 changes that worsen them. Two positive observations about the review surface itself (Amendment discipline codified; architecture docs as security multiplier) feed forward into the B1 / B2 review cadence. Surface ready for B1 work to land on top.
