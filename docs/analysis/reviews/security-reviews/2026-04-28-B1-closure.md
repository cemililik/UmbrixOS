## 1. Capability correctness

Adversarial frame: *can a caller invoke any privileged operation introduced by T-012 without holding the capability the security model requires? Does the new IRQ-delivery path widen any existing capability check?*

- **Phase A + B0 invariants intact.** All "OK" findings from the [B0 closure security review](2026-04-27-B0-closure.md) re-verified against the post-PR-#10 source. No regression in `cap_copy` / `cap_derive` / `cap_revoke` / `cap_drop` / `cap_take` / `ipc_send` / `ipc_recv` / `ipc_send_and_yield` / `ipc_recv_and_yield`. T-012 added no new capability operation; the GIC, vector table, and timer arm/cancel are kernel-internal interfaces (see §4 below for the kernel-mode discipline argument that this is correct).
- **`Timer::arm_deadline` / `cancel_deadline` are not capability-gated in v1.** OK — and deliberate. The trait sits at the HAL surface; only the kernel itself (today: no caller; future: a `time_sleep_until` syscall in B5+) calls these. The HAL layer is below the capability layer — the same decision as `Console::write_bytes` (no capability check at the HAL level; the kernel's userspace-exposed `console_write` syscall will check `ConsoleCap` once syscalls land in B5). Equivalent reasoning: a future `time_sleep_until` syscall will check the caller's `TimerCap` (or equivalent) before forwarding to `Timer::arm_deadline`. **Adversarial probe:** can a hostile task today reach `arm_deadline`? No — there is no caller. The IPC demo's `task_a` / `task_b` make no Timer-trait call. Idle's body uses `wait_for_interrupt`, which goes through `Cpu::wait_for_interrupt` (not Timer). The first userspace-reachable arm_deadline path is gated by Phase B5's syscall-capability check.
- **`IrqController::enable` / `disable` / `acknowledge` / `end_of_interrupt` are not capability-gated.** OK — also deliberate. These operate on hardware lines, not on capabilities; they correspond to `RWX` page-table ops in spirit, not to capability-table ops. The kernel calls them from within trusted contexts: `arm_deadline` / `cancel_deadline` for enable/disable, `irq_entry` for acknowledge/EOI. **Adversarial probe:** can a hostile caller reach `enable(IrqNumber(arbitrary))`? Today the surface is `&dyn IrqController` exposed to the BSP via the `GIC` static. Reaching it from outside the BSP would require the kernel to forward a userspace request to the IRQ controller, which v1 does not do. Future userspace IRQ delivery (the `IrqCap` pattern from `docs/architecture/exceptions.md` §"Trade-offs → No userspace IRQ delivery") is Phase C work; the security model already names that as the capability-system path.
- **GIC `enable` / `disable` range assertion.** OK — `assert!(irq.0 < GIC_MAX_IRQ)` (1020) at [`bsp-qemu-virt/src/gic.rs:317-322` and `:343-348`](../../../../bsp-qemu-virt/src/gic.rs) gates the offset math against ARM IHI 0048B §4.3.2's architectural maximum. An out-of-range `IrqNumber` would compute an MMIO offset outside the distributor window even with saturating arithmetic; the assertion converts a kernel-internal contract violation into a loud, named panic before MMIO. The panic message includes the bad value and the architectural cap, useful for triage. **Adversarial probe:** is the assertion *before* any side effect? Yes — it runs before `write_distributor`, so a failure leaks no MMIO state. **No reachable hostile caller in v1** (only `arm_deadline` / `cancel_deadline` call `gic.enable` / `disable`, and they pass `IrqNumber(27)` — a constant); the assertion is defence-in-depth against future regressions.
- **`acknowledge` returns `None` on spurious INTID 1023.** OK — [`gic.rs:369-376`](../../../../bsp-qemu-virt/src/gic.rs) reads `GICC_IAR`, masks the bottom 10 bits, and folds `1023` to `None`. This is the trait contract. **Adversarial probe:** can the spurious value leak as a real `IrqNumber(1023)` to a downstream caller? No — the mask-then-test path is exclusive. A future GICv3 implementation would extend this to also fold `1023` because the architecture preserves the spurious-INTID convention.
- **`end_of_interrupt` writes the raw IRQ ID.** OK — [`gic.rs:378-384`](../../../../bsp-qemu-virt/src/gic.rs) writes `irq.0` to `GICC_EOIR` without further validation. **Adversarial probe:** what if a caller passes an `IrqNumber` that did *not* come from `acknowledge`? GICv2 EOIR ignores writes that don't match a currently-active priority level; the write is silently dropped. No state corruption, no privilege escalation. The trait contract documents the pairing rule ("`end_of_interrupt` takes the number from `acknowledge`"); this is a kernel-internal discipline, not a security boundary.

## 2. Trust boundaries

Adversarial frame: *what new boundaries did T-012 introduce, and what untrusted bytes (or values) cross them?*

- **No userspace/kernel boundary added.** OK — same as B0. Syscalls remain Phase B5+ work; T-012 introduced no new in-code boundary that crosses privilege levels.
- **GIC MMIO surface (UNSAFE-2026-0019).** OK — every read/write goes through `core::ptr::read_volatile` / `write_volatile` against typed offsets within the distributor (`0x0800_0000`) and CPU interface (`0x0801_0000`) windows. **Adversarial probe:** can the offset arithmetic escape its window? Distributor offsets are bounded by:
  - `GICD_CTLR` (0x000), `GICD_TYPER` (0x004): fixed.
  - `GICD_ISENABLER` / `GICD_ICENABLER` / `GICD_IPRIORITYR` / `GICD_ITARGETSR`: indexed by `n` or `byte_idx`. The bounds in `gic.init`'s loops are derived from `GICD_TYPER`'s bottom 5 bits (max value 0x1F → 32 line groups → 1024 IRQ lines). The architectural cap is 1024; the loop condition is `byte_idx < irq_count` so for `irq_count = 1024` the *last* `byte_idx` written is 1020, giving max offset `GICD_ITARGETSR_BASE + 1020 = 0x800 + 0x3FC = 0xBFC` (the next iteration's 1024 exits the loop, giving a one-past-end of `0xC00`). Distributor windows are at minimum 4 KiB on every GICv2 implementation; the last word write at `0xBFC` is well within. ✓
  - `GICD_ISENABLER + (n / 32) * 4` and `GICD_ICENABLER + (n / 32) * 4` for `n < GIC_MAX_IRQ = 1020`: largest offset is `0x180 + 124 = 0x1FC` (clear-enable) and `0x100 + 124 = 0x17C` (set-enable). Both within the window. ✓
  - CPU interface offsets are fixed: 0x000 / 0x004 / 0x00C / 0x010. ✓
- **GIC `init` runs with `DAIF.I` masked.** OK — [`bsp-qemu-virt/src/main.rs:554-585`](../../../../bsp-qemu-virt/src/main.rs) sequences "MSR VBAR_EL1 → GIC construct → GIC init" before "MSR DAIFClr, #0x2" at line 596. **Adversarial probe:** if `init` faulted (bad MMIO base), is the failure caught? Yes — `VBAR_EL1` is installed before `init`, so any sync exception in `init`'s MMIO would land at `tyrne_vectors+0x200`, which trampolines to `panic_entry(class=0, ESR_EL1)`. The panic message would surface the fault loudly. ✓
- **EL1 vector table install via `MSR VBAR_EL1` (UNSAFE-2026-0020).** OK — the address is read via `core::ptr::addr_of!(tyrne_vectors) as u64` (no deref of the zero-byte `extern "C" static u8`), written to `VBAR_EL1` with `ISB` for instruction-stream sync. **Adversarial probe:** is the 2 KiB alignment enforced? Two layers: (a) `vectors.s` declares `.balign 2048` at the start of `.text.vectors`; (b) [`linker.ld`](../../../../bsp-qemu-virt/linker.ld) places `.text.vectors` after a `. = ALIGN(2048)` boundary inside `.text`. Either alone is sufficient; both is defence in depth. ARM ARM §D11.2 requires the bottom 11 bits of `VBAR_EL1` to be zero, otherwise the `eret`-from-handler path lands at the wrong byte offset.
- **IRQ trampoline frame layout (UNSAFE-2026-0020).** OK — the asm `stp` sequence in [`bsp-qemu-virt/src/vectors.s:114-128`](../../../../bsp-qemu-virt/src/vectors.s) writes through fixed offsets `0x00..0xA0` mirroring the field order of `TrapFrame` in [`bsp-qemu-virt/src/exceptions.rs:43-70`](../../../../bsp-qemu-virt/src/exceptions.rs). PR #10 review-round-2 added a compile-time guard `const _: () = assert!(core::mem::size_of::<TrapFrame>() == 192)` immediately after the struct ([`exceptions.rs:72-77`](../../../../bsp-qemu-virt/src/exceptions.rs)), so any drift between asm and `repr(C)` fails the build before it can ship. **Adversarial probe:** can a `repr(C)` reordering by a future Rust toolchain break the layout? No — `repr(C)` is a stable Rust ABI; field offsets are guaranteed lexicographic per the C ABI conventions for structs of `[u64; 2]`-typed fields. The compile-time size guard is the empirical backstop; if it ever fired, the build would not produce a kernel image.
- **`tyrne_vectors_end` symbol exported but unused by linker.ld.** N/A from a security perspective — declared in [`vectors.s:196-197`](../../../../bsp-qemu-virt/src/vectors.s) as a future-tools convenience. Recommend (non-blocking) that a future tooling pass — once `cargo-vet` / `geiger` integration adds region-bound checks per UNSAFE-policy §3 — uses this symbol to verify the vector region's audit-log claims.
- **Unhandled-exception trampoline lumps sync/FIQ/SError as `class = 0`.** OK — the `mov x0, #0` in [`vectors.s:165`](../../../../bsp-qemu-virt/src/vectors.s) hard-codes the class for any of {sync, FIQ, SError on any mode-class}. The diagnostic value is preserved by the paired `mrs x1, esr_el1`: the panic message at [`exceptions.rs:222-224`](../../../../bsp-qemu-virt/src/exceptions.rs) interpolates `ESR_EL1` raw, which decodes the actual class bits[31:26] and the syndrome bits[24:0]. The trampoline-side simplification is the deliberate v1 choice documented at [`exceptions.rs:74-78`](../../../../bsp-qemu-virt/src/exceptions.rs) ("the goal is to surface 'something hit an unhandled vector' loudly, not to provide rich diagnostics in v1"). **Adversarial probe:** does the lossy class encoding hide a security-relevant condition? No — the panic itself is the security signal; richer decoding would help triage but does not change the fact that an unhandled exception is fatal.
- **Vector entry padding fills with `0x00` = `udf` on aarch64.** OK — `.balign 0x80` after each `b <label>` instruction pads with the assembler's default fill byte. On aarch64 GAS that is `0x00`, which decodes as the architecturally-defined undefined instruction `udf #0x0` — a CPU that somehow lands inside the padding faults loudly with a sync exception, which routes back to `panic_entry`. Self-stabilising. **Adversarial probe:** could a future linker change pad with garbage? Not without changing the assembler / linker explicitly. The `0x00` default has been GAS's behaviour for the entire aarch64 backend's history.

## 3. Memory safety

Adversarial frame: *can a hostile caller, or an unexpected interrupt arrival, cause any of the three new audited unsafe regions (UNSAFE-2026-0019 / 0020 / 0021) to violate the invariants it claims?*

- **Kernel crate's `unsafe` surface unchanged by T-012.** OK — `grep -rE "^\s*unsafe\s*\{" kernel/src/` returns 15 inner `unsafe { ... }` blocks, all in `kernel/src/sched/mod.rs` (the raw-pointer scheduler bridge `unsafe fn`s' bodies, mandatory under `#![deny(unsafe_op_in_unsafe_fn)]`). These are pre-existing per ADR-0021 / T-006 (B0); T-012 added zero new kernel-crate `unsafe` regions. The new surface is BSP-internal (`bsp-qemu-virt/src/{gic,exceptions,cpu}.rs` plus `bsp-qemu-virt/src/{vectors,boot}.s`). The kernel crate has no IRQ-side `unsafe`; the IRQ infrastructure lives entirely in the BSP layer per the trait-impl shape ADR-0011 / 0024 + the audit-log entries this review covers.
- **UNSAFE-2026-0019 — `QemuVirtGic` MMIO.** OK — the entry's Operation field accurately matches the shipped surface: const `unsafe fn new` (no MMIO; stores bases), `unsafe fn init` (boot-time programming sequence), four `IrqController` trait method bodies (single-register MMIO each), four private MMIO helpers (`read_distributor` / `write_distributor` / `read_cpu_interface` / `write_cpu_interface`), and the `unsafe impl Send` / `unsafe impl Sync` markers. Every MMIO call uses `core::ptr::read_volatile` / `write_volatile`. **Adversarial trace of a single `enable(IrqNumber(27))` call:**
  - `assert!(27 < 1020)` ✓ passes.
  - `n = 27`, `reg_offset = 0x100 + (27 / 32) * 4 = 0x100`.
  - `bit = 1u32 << (27 % 32) = 0x0800_0000`.
  - `write_distributor(0x100, 0x0800_0000)` → `write_volatile((0x0800_0000 + 0x100) as *mut u32, 0x0800_0000)` = write to `GICD_ISENABLER<0> + 0x100` = sets enable-bit 27.
  - GICD_ISENABLER<0> covers IRQs 0..32 — within window.
  No bounds violation; no aliasing (single-core; GIC MMIO is not Rust-memory-aliased anyway). ✓
- **UNSAFE-2026-0020 — Vector table install + asm trampolines.** OK — the entry's three contiguous capabilities (vector-table assembly; `MSR VBAR_EL1`; trampolines + Rust handlers) match the shipped surface. The `unsafe extern "C" fn irq_entry` and `panic_entry` signatures (post PR #10 round 1 commit `d820a88`) make the trampoline-supplied frame-validity precondition visible at the type level. **Adversarial trace of an IRQ taken while running kernel code:**
  - Hardware: `DAIF.I = 0` cleared; pending IRQ at GIC; CPU takes the exception; `ELR_EL1 ← interrupted PC`, `SPSR_EL1 ← interrupted PSTATE` (DAIF auto-masked by hardware on entry), PC ← `VBAR_EL1 + 0x280`.
  - Trampoline: `sub sp, sp, #192` allocates frame; 11 `stp` save x0..x18, x30; `mrs x0, elr_el1`, `mrs x1, spsr_el1`, `stp x0, x1, [sp, #0xA0]`; `mov x0, sp`; `bl irq_entry`.
  - `irq_entry` body: take `&QemuVirtGic` from the static GIC cell (UNSAFE-2026-0019 audit cross-reference); call `gic.acknowledge()`; spurious → `compiler_fence(SeqCst); return`; non-spurious + IRQ 27 → `MSR CNTV_CTL_EL0, 0b10` (UNSAFE-2026-0021 cross-reference) + `gic.end_of_interrupt(IrqNumber(27))`; non-27 → panic.
  - Trampoline restore: 11 `ldp` restore x0..x18, x30; `ldp` reads ELR/SPSR; `msr elr_el1, x0`; `msr spsr_el1, x1`; `add sp, sp, #192`; `eret`.
  - Hardware: `eret` consumes ELR_EL1 + SPSR_EL1; PC ← interrupted PC; PSTATE.DAIF ← interrupted DAIF (typically `I = 0` again).
  No aliasing of any kernel static; no `&mut` to `Scheduler<C>` or any arena; the only momentary `&` is to `QemuVirtGic` (immutable, single-instance) and the `&mut TrapFrame` is never materialised in v1's `irq_entry` body. ADR-0021's 2026-04-28 Amendment is the canonical record of this discipline; v1 vacuously satisfies it.
- **UNSAFE-2026-0021 — Generic-timer compare-register writes.** OK — the entry's three architected MSRs (`CNTV_CVAL_EL0`, `CNTV_CTL_EL0 = 0b01`, `CNTV_CTL_EL0 = 0b10`) match the shipped surface in [`bsp-qemu-virt/src/cpu.rs:484-553`](../../../../bsp-qemu-virt/src/cpu.rs) plus the `CNTV_CTL_EL0 = 0b10` write inside `irq_entry`'s timer-IRQ dispatch. Each MSR is `options(nostack, nomem)`-correct (system-register writes, no memory touch). **Adversarial probe — race window between `MSR CNTV_CVAL_EL0` and `MSR CNTV_CTL_EL0` in `arm_deadline`:** if the timer was *already* armed when `arm_deadline` ran (`ENABLE = 1`) and `CNTVCT >= new_CVAL` between the two writes, the timer could fire on the new CVAL. The IRQ would route through `irq_entry` which masks the timer + EOIs — clean from a memory-safety standpoint. From a *correctness* standpoint, this is a "spurious-but-clean" delivery, identical to the cancel→delivery race; the next `arm_deadline` re-arms correctly. **No caller in v1.** The first caller will be a future `time_sleep_until` syscall in B5+ that calls `cancel_deadline` first if needed; the security review for that syscall's introduction will revisit. Forward-only flagged item: a future preemption / `time_sleep_until` arc should mask `DAIF.I` around the CVAL+CTL pair if back-to-back arming is a hot path.
- **UNSAFE-2026-0014 commit-`28c5ce9` Amendment — IRQ-handler frame as future site of momentary-`&mut` pattern.** OK — the Amendment names `irq_entry` as a future site; v1's body has no live citation because it never materialises a `&mut Scheduler<C>` / `&mut EndpointArena` / `&mut IpcQueues` / `&mut CapabilityTable`. The Amendment is the *forward-discipline* record, not a current site. **Adversarial probe:** if a future arc adds `sched::on_timer_irq` to `irq_entry`, what does the safety contract look like? ADR-0021's 2026-04-28 Amendment §three points lay it out: (1) signature stays `unsafe extern "C" fn irq_entry(_frame: *mut TrapFrame)` — no `&mut Scheduler<C>` parameter; (2) momentary `&mut` materialised inside a scope ending strictly before `eret`; (3) per-site `// SAFETY:` comment + UNSAFE-log Amendment recording the activation. The discipline is structurally identical to the cooperative-bridge case; v1's vacuous satisfaction is the easy half.
- **`TrapFrame` compile-time size guard.** OK — `const _: () = assert!(core::mem::size_of::<TrapFrame>() == 192);` at [`exceptions.rs:72-77`](../../../../bsp-qemu-virt/src/exceptions.rs). This is the strongest possible safety net for the asm/Rust ABI invariant; a future field-reordering or padding change fails the build. **Adversarial probe:** could a future Rust toolchain change `repr(C)` semantics to insert tail padding that breaks the assertion? Extremely unlikely (`repr(C)` for a struct of `[u64; 2]` fields is layout-stable per the C ABI), and even if it did, the build would fail loudly rather than silently miscompile. ✓
- **`compiler_fence(SeqCst)` in spurious-IRQ early-return.** OK (defensive, non-load-bearing) — [`exceptions.rs:147-153`](../../../../bsp-qemu-virt/src/exceptions.rs). The early-return path is the only place a `compiler_fence` is needed because the dispatch-and-EOI path already includes a `gic.end_of_interrupt` call which compiles to a `write_volatile`, which itself is a compile-time barrier. The spurious-return path has no volatile op, so the fence guarantees the compiler does not reorder the `return` against any subsequent (re-entrant) instruction. **Adversarial probe:** is the fence load-bearing? Not in v1 — `irq_entry`'s body is straight-line, no surrounding code that could be reordered. Defence in depth for forward arcs that may add post-`return` work.
- **Forward-flagged item — `cancel_deadline` ordering race window.** OK in v1 (no caller); recorded for the same future-arc audience as the `arm_deadline` CVAL+CTL race below. [`bsp-qemu-virt/src/cpu.rs:532-554`](../../../../bsp-qemu-virt/src/cpu.rs)'s `cancel_deadline` writes `CNTV_CTL_EL0 = 0b10` (mask source) at Step 1 and calls `gic.disable(TIMER_IRQ)` at Step 2; between the two, an IRQ that was *already pending at the GIC distributor* (queued before Step 1's source-mask took effect) can deliver to the CPU interface and route through `irq_entry`. The handler masks `CNTV_CTL_EL0` (already masked — idempotent) and EOIs at the GIC; this is the same "spurious-but-clean" delivery shape as the `arm_deadline` race below. Memory-safety: untouched. Correctness: the IRQ fires once after `cancel_deadline` returns, which a caller that *just* cancelled would not expect. **Mitigation for any future caller (B5+ `time_sleep_until` syscall):** the symmetric advice as for `arm_deadline` — mask `DAIF.I` around the `MSR CNTV_CTL_EL0` + `gic.disable` pair if back-to-back cancel/arm patterns are a hot path; or accept the spurious-but-clean delivery and design the handler to absorb it (current shape). The security review for the first user-reachable caller's introduction will revisit.

## 4. Kernel-mode discipline

Adversarial frame: *does any new code path violate the principle that kernel-mode work be minimal, deterministic, and panic-only-on-impossibility?*

- **`irq_entry` is panic-free on the structurally-reachable paths.** OK — only the "any other IRQ" branch panics ([`exceptions.rs:185-192`](../../../../bsp-qemu-virt/src/exceptions.rs)), and that branch is structurally unreachable in v1 because the GIC enables only `IrqNumber(27)`. The panic is the right shape: it indicates kernel-state corruption upstream (e.g., the GIC delivered an IRQ whose number we never enabled), and surfacing loudly rather than silently is correct kernel discipline. **Adversarial probe:** can a hostile arrangement of IRQ delivery reach the panic? Not without first mis-configuring the GIC, which itself requires kernel access — the `enable` calls that route IRQs to the CPU interface are kernel-internal. A misbehaving emulator that delivered a phantom IRQ would also reach the panic; loud failure is preferred to silent recovery here.
- **`irq_entry`'s ack-and-ignore body is constant-time.** OK — straight-line: read GICC_IAR (1 volatile read), branch on spurious (1 cmp + 1 b), branch on IRQ ID (1 cmp + 1 b), `MSR CNTV_CTL_EL0` (1 instruction), `gic.end_of_interrupt` (1 volatile write), return. ~10 instructions of dispatch logic plus the 31 instructions of trampoline save/restore — bounded, predictable, no allocation, no scheduler touch. Matches the [`docs/architecture/exceptions.md`](../../../architecture/exceptions.md) §Invariants → "ISR minimality" claim.
- **No new heap allocation.** OK — T-012 added zero `alloc::*` imports, zero `Box::new`, zero `Vec::new`. Kernel remains heap-free per ADR-0001. The 192-byte `TrapFrame` is on the kernel stack (allocated by `sub sp, sp, #192`); not a heap allocation.
- **No unbounded loops in ISR.** OK — `irq_entry`'s body has no loop. The dispatch is straight-line.
- **No new kernel panic on a *hot* path.** OK — the new panic in `irq_entry` is the structurally-unreachable "unhandled IRQ" guard; the GIC range-assertion panics in `enable` / `disable` are also defensive against kernel-internal regressions, not user input. No new userspace-reachable panic.
- **Idle's WFI activation.** OK — [`bsp-qemu-virt/src/main.rs:255-286`](../../../../bsp-qemu-virt/src/main.rs)'s `idle_entry` body uses `cpu.wait_for_interrupt() + yield_now`. `wait_for_interrupt` issues `WFI`, which suspends the core until any unmasked IRQ raises a wake. **Adversarial probe — what if no IRQ ever fires?** WFI hangs the core forever. In v1, the cooperative IPC demo's two application tasks are permanently `Ready`, so the FIFO never reaches `idle_entry` — WFI is structurally unreachable in the demo path. A future caller that arms `arm_deadline` from a task body would provide the wake source; until then, the v1 deadlock-style state is "all tasks Ready, idle never runs", not "WFI hung". Documented at [`main.rs:268-274`](../../../../bsp-qemu-virt/src/main.rs). The typed `SchedError::Deadlock` from ADR-0022 is reachable only if the BSP did not register an idle task at all (which v1's `kernel_entry` does at line 700).
- **GIC + DAIF unmask ordering in `kernel_entry`.** OK — [`main.rs:548-598`](../../../../bsp-qemu-virt/src/main.rs) sequences "VBAR_EL1 install → GIC construct + init → DAIF unmask" in that order. **Adversarial probe — what would re-ordering cause?** (a) GIC init before VBAR install: a fault in init would jump to VBAR_EL1's reset value (0x0 typically) and silently hang. (b) DAIF unmask before GIC init: an IRQ pending from firmware/emulator could fire before the GIC's CPU interface is enabled, but the GIC wouldn't deliver it because the CPU interface is disabled — at worst, a missed legacy-firmware IRQ. The current order is the safe-by-construction one. ✓
- **Scheduler does not touch the new IRQ surface.** OK — `kernel/src/sched/mod.rs` was unchanged by T-012. The scheduler is unaware of IRQs in v1; the cooperative-only scheduling model is preserved. The future preemption arc will introduce a `sched::on_timer_irq` hook per the ADR-0021 2026-04-28 Amendment.
- **DAIF.I unmask is the only kernel write to DAIF in `kernel_entry`.** OK — `DAIFClr, #0x2` clears bit 1 (`I`) only; bits 0 (`F`), 2 (`A`), 3 (`D`) stay masked. v1 has no FIQ source, no SError-recoverable scenario, no debugger; the unchanged-mask defaults are correct. A future driver that needs FIQ would unmask `F` separately under its own ADR.

## 6. Secrets and logging

- **Boot-time logging.** OK — T-012 added no new console writes besides the existing T-009 timer banner (`tyrne: timer ready (... Hz, resolution ... ns)`). ## 8. Threat-model impact

Adversarial frame: *did the T-012 arc reshape what the system defends against, or what it concedes to attackers?* Three load-bearing shifts plus three positive observations.

**T-012 promotes "the kernel can take an interrupt" from a hoped-for capability to a structural property.** Before T-012, the kernel ran with `DAIF.I` masked from the reset vector onward (per ADR-0024 + the K3-12 boot-checklist rule). Any unmasked-IRQ-arrival was a kernel-state corruption signal, but the kernel had no infrastructure to handle it — vector table un-installed, GIC unconfigured. Post-T-012, the kernel actively drives `DAIF.I` unmasked after `VBAR_EL1` install + `GIC init`. The threat-model implication: the kernel's "we can be interrupted" assumption is now an enforced invariant rather than a "we hope nobody interrupts us before we install the vector table" hope. A misbehaving emulator that delivered a stray IRQ during boot was previously a silent risk surface (jump to `VBAR_EL1` reset value); T-012 closes that risk by installing the vector table before the unmask.

**T-012 widens the kernel's `unsafe` surface from 18 to 21 audited entries.** UNSAFE-2026-0019 (GIC v2 MMIO, ~150 lines of Rust + asm-volatile MMIO), UNSAFE-2026-0020 (vector table install + asm trampolines, ~220 lines of asm + ~100 lines of Rust), and UNSAFE-2026-0021 (generic-timer compare-register writes, ~50 lines of Rust). The threat-model implication: the `unsafe` review surface is now ~16 % larger. The mitigation is the discipline: every block has a `// SAFETY:` triplet (PR #10 round-2 enforced this rigorously); every audit entry has full rejected-alternatives text; the introducing-commit boundary rule (UNSAFE-2026-0017's "Discipline note") applies to the new entries. The net change is *more* enforcement infrastructure, not just more `unsafe`.

**ADR-0021's IRQ-handler aliasing-discipline Amendment (2026-04-28) extends the no-`&mut`-across-switch rule to a second axis.** Pre-T-012, the rule covered the cooperative-bridge case only (`ipc_send_and_yield` / `ipc_recv_and_yield` / `yield_now` / `start`). Post-T-012, it explicitly covers the IRQ-handler frame: any future `irq_entry` body that wants to mutate scheduler state must follow the same momentary-`&mut` pattern. The threat-model implication: the kernel's memory-safety story now has a *single* aliasing discipline that covers both cooperative yields and preemptive IRQ-driven entry points. A future preemption arc inherits the discipline rather than rediscovering it. This is a strict improvement in what the kernel can be defended against — a rediscovery would have produced a parallel invariant; the Amendment produces one shared invariant.

**Threat-model items inherited from Phase A / B0 (unchanged severity).** Both predate T-012; neither is worsened or improved by it. ADR-0023's deferred-acceptance path remains the planned closure for the cross-table case:

- **Cross-table revocation gap** — tracked as ADR-0023 (deferred); re-examine when the first multi-task server arc starts in B3–B6.
- **Generation overflow (`u32`)** — tracked as a long-running-service concern; re-examine in B-late phases.

**Positive observations about the review surface itself** (not threat-model shifts in the kernel; observations about how this review and future reviews will work):

- **Architecture doc landed *with* the code, not as a follow-up.** [`docs/architecture/exceptions.md`](../../../architecture/exceptions.md) — drafted first per the design-first arc the [B0 closure security review §8](2026-04-27-B0-closure.md) recommended — synthesises ADR-0010 / 0011 / 0021 / 0022 / 0024 with two Mermaid diagrams, an implementation map mapping each T-012 step to its commit, and a §"IRQ-handler interaction with the raw-pointer scheduler bridge" subsection that pre-states the ADR-0021 Amendment's reasoning. The review surface is materially smaller because the synthesis is already on disk; recommendation that B2's MMU activation arc continues this pattern.
- **PR #10 review-fix sweeps closed 19 findings + 2 nitpicks before merge.** Two independent rounds (`d820a88` round-1, `5c9cf06` round-2) caught: missing `unsafe` on `irq_entry` / `panic_entry` signatures (round-1), `GIC_MAX_IRQ` range assertion missing (round-1), `ns_to_ticks` floor-vs-ceiling rounding bug at non-divisor frequencies (round-1), full SAFETY-comment triplets needed for the inline blocks (round-2), `TrapFrame` compile-time size guard (round-2), `ns_to_ticks_rounds_up_on_subtick` test (round-2). Each finding is a class of bug the review-fix-round cadence catches before merge — the discipline pays for itself.
- **`unsafe` audit-log entries shipped with explicit `Pending QEMU smoke verification` status notes.** Each of UNSAFE-2026-0019 / 0020 / 0021 carries a status note flagging that the kernel-build-clean half is verified but the actual MMIO discipline + IRQ-take + dispatch path is observable only in QEMU smoke. The maintainer will lift these via append-only Amendment after running the smoke. This is the right shape — honest about what hasn't been verified, with a clear closure path. The introducing-commit boundary still applies (the *body* of each entry was locked at the introducing commit; only an Amendment flips the verification-status sub-claim).

## Verdict

**Approve.** All eight axes pass; no high-severity findings. The three threat-model shifts T-012 introduces (IRQ delivery as structural property; expanded `unsafe` surface with proportionally expanded audit/discipline infrastructure; IRQ-handler aliasing discipline as a unified second-axis rule) are net improvements to what the kernel can be defended against. Previously-flagged Phase-A / B0 inheritance items (cross-table revocation, `u32` generation overflow, K3-9 `Capability::Debug` redaction) are tracked at original severity with no T-012 changes that worsen them. Three positive observations about the review surface itself (design-first architecture doc as security multiplier, review-fix-round cadence closing 19+2 findings pre-merge, audit-log status notes as explicit verification-pending markers) feed forward into the B2 review cadence.

**Two forward-flagged items, both non-blocking** (recorded in §3 above; both are benign in v1 because no caller exists):

1. The `arm_deadline` race window between `MSR CNTV_CVAL_EL0` and `MSR CNTV_CTL_EL0`.
2. The `cancel_deadline` race window between `MSR CNTV_CTL_EL0 = 0b10` (mask source) and `gic.disable(TIMER_IRQ)`.

The first userspace-reachable caller (Phase B5+ `time_sleep_until` syscall) should mask `DAIF.I` around the relevant register-write pair if back-to-back arming or cancel/arm patterns are a hot path; alternatively, the handler can absorb the resulting spurious-but-clean delivery (current shape) without correctness or memory-safety impact. The next security review covering that arc revisits.

**Pre-closure work-items (gating B1 → Done at the milestone level rather than at the task level):**

1. **Maintainer-side QEMU smoke verification of T-012's deliberate-deadline path.** UNSAFE-2026-0019 / 0020 / 0021 status notes are "Pending QEMU smoke verification at the maintainer's first opportunity"; lifted via append-only Amendment after the smoke run. The smoke trace should reproduce the A6 + T-009 + T-013 baseline (kernel boots, prints greeting + timer banner + idle's structurally-unreachable WFI section + IPC demo + boot-to-end elapsed) and additionally exercise an `arm_deadline(now + 100ms)` from a debug task (deferred until a debug task is added; not a v1-IPC-demo path) — the deferred half is documented in T-012's DoD as maintainer-side work.
2. **`cargo +nightly miri test --workspace --exclude tyrne-bsp-qemu-virt`.** Same disclaimer T-013 used; pinned for the maintainer's CI / R6 GitHub Actions skeleton.

Surface ready for B2 (MMU activation, kernel-half mapping) work to land on top of the now-IRQ-aware kernel.
